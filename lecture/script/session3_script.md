## 1-1. 문제 제기 (3분)

상황을 하나 그려보겠습니다.

여러분이 어떤 제조 공장의 안전 담당자라고 해볼게요. 라인마다 CCTV가 설치되어 있고, 작업자가 안전모를 썼는지, 위험 구역에 들어가지 않았는지를 실시간으로 모니터링해야 합니다. 인력으로 24시간 보는 건 비효율적이고 사람은 졸음이 옵니다. 그래서 AI에게 맡기고 싶다, 이게 출발점이죠.

가장 단순한 발상은 이렇습니다. **모든 영상을 GPU 서버로 보내서 분석하자.** 클라우드 서버에 강력한 GPU를 띄워 두고, 공장 카메라 영상을 인터넷을 통해 보내고, 결과를 다시 받아오는 거죠. 듣기엔 깔끔합니다.

하지만 실제로 해보면 문제가 세 가지 터집니다.

첫째, **지연 시간**입니다. 영상이 인터넷을 왕복하면 0.5초에서 2초 정도가 걸립니다. 그런데 사고는 0.5초 안에도 일어납니다. 이미 손이 끼인 후에 경보가 울리는 거죠. 안전 모니터링이라는 본래 목적이 망가집니다.

둘째, **네트워크 비용**입니다. 30FPS 풀HD 영상 한 채널이 시간당 약 1.5GB씩 만들어집니다. 카메라가 50대라고 하면 하루에 수 TB의 데이터가 외부로 빠져나갑니다. 비용도 비용이지만, 회사의 전용회선이 영상 트래픽에 잠겨버립니다.

셋째, **보안**입니다. 공장 내부 영상은 일종의 영업 비밀입니다. 라인 배치, 공정 순서, 인력 동선이 다 찍혀 있죠. 이걸 외부 서버로 전송하는 건 많은 회사가 정책적으로 막혀 있습니다.

해결책의 방향은 명확합니다. **엣지 컴퓨팅**, 즉 카메라 옆에 작은 컴퓨터를 두고 거기서 분석을 끝내는 거죠. Jetson Nano나 Raspberry Pi 같은 기기를 떠올리시면 됩니다. 영상은 로컬에서 처리되고, 결과 한 줄, 예를 들어 "12번 라인 안전모 미착용 의심"만 서버로 전송되는 구조입니다. 지연 없고, 네트워크 부담 없고, 보안도 유지됩니다.

자, 그런데 여기서 진짜 어려운 질문이 시작됩니다.

**GPU도 없는, 손바닥만 한 컴퓨터에서 어떻게 실시간으로 객체 탐지가 가능한가?**

정확도를 충분히 유지하면서 속도를 어디까지 끌어올릴 수 있는가? 이 트레이드오프를 어떻게 풀 것인가? 오늘 섹션 1 전체가 이 질문에 대한 답입니다.

## 1-2. 이론 (13분)

### ① Object Detection의 발전: 왜 YOLO인가 (4분)

*Practical Machine Learning for Computer Vision* 4장에 나오는 이야기부터 시작하겠습니다.

객체 탐지, Object Detection의 역사는 크게 두 갈래로 발전해 왔습니다.

첫 번째 갈래가 **2단계 탐지기**, 영어로 Two-stage detector입니다. R-CNN 계열이 대표적이고, R-CNN, Fast R-CNN, Faster R-CNN, Mask R-CNN으로 이어지죠. 이 방식의 핵심 아이디어는 이름 그대로 "두 단계로 나눠서 한다"입니다.

1단계에서는 이미지의 **어디를 봐야 할지**를 먼저 정합니다. Region Proposal이라고 부르는 단계입니다. 후보 영역을 수백, 수천 개 만들어내죠. 2단계에서는 그 각각의 후보 영역을 잘라서 **무엇이 있는지** 분류합니다. 정확도는 높습니다. 후보 영역마다 정성스럽게 분류하니까요. 하지만 속도가 느립니다. 초당 2프레임에서 10프레임 정도, FPS로 표현하면 2~10 FPS입니다. CCTV가 보통 30FPS인데 그 1/10도 못 따라가는 거죠. **실시간으로는 사용할 수 없습니다.**

두 번째 갈래가 **1단계 탐지기**, Single-stage detector입니다. YOLO, SSD, RetinaNet이 여기에 속합니다. 이름 그대로 단 한 번의 추론으로 위치와 클래스를 동시에 예측합니다. YOLO라는 이름이 그래서 "You Only Look Once", 한 번만 본다는 뜻이죠. 책에서는 "이미지를 한 번만 보고 모든 객체를 동시에 예측한다"고 표현합니다. 정확도는 2단계 방식보다 약간 낮습니다. 하지만 속도는 비교가 안 됩니다. 30 FPS에서 100 FPS 이상까지 나옵니다.

제조 현장에서 실시간 모니터링이라는 요구를 충족시키려면 답은 거의 정해져 있습니다. **YOLO 계열이 사실상의 표준입니다.** 정확도 1~2%를 양보하고 속도 10배를 얻는 트레이드오프죠.

물론 YOLO가 한계도 있습니다. *Practical ML for Computer Vision* 4장에서 명시적으로 언급하는 단점이 두 가지입니다. 하나는 한 그리드 셀당 하나의 클래스만 예측한다는 점입니다. 작은 새들이 떼지어 모여 있는 상황 같은 건 잘 못 잡습니다. 또 하나는 마지막 feature map 하나만 쓰니까 작은 객체 탐지가 약하다는 점이죠. 이 한계를 보완하기 위해 RetinaNet, FPN 같은 후속 아키텍처가 나왔다는 건 뒤에 잠깐 짚고 넘어가겠습니다.

### ② YOLOv8 구조 직관 (5분)

오늘 우리가 실제로 쓸 모델은 YOLOv8입니다. v1부터 v8까지 발전한 결과물이죠. 구조를 직관적으로 이해할 수 있게 세 부분으로 쪼개 보겠습니다.

첫 번째는 **백본**, Backbone입니다. YOLOv8에서는 CSPDarknet이라는 구조를 씁니다. 백본의 역할은 단순합니다. 이미지에서 **특징을 뽑는 것**이죠. 텍스처, 모서리, 형태, 색상 같은 시각적 단서들을 점점 더 추상화된 표현으로 압축해 나갑니다. 입력은 640×640 픽셀의 RGB 이미지, 출력은 이 이미지를 압축한 다층 특징 맵입니다.

두 번째는 **넥**, Neck입니다. FPN과 PAN을 조합한 구조죠. 여기서 FPN은 Feature Pyramid Network의 약자입니다. *Practical ML for Computer Vision* 4장에 잘 정리되어 있는데, 핵심은 "다양한 크기의 특징을 같이 다룬다"는 겁니다. CNN을 통과시키면 깊은 층의 특징 맵은 의미는 풍부하지만 공간 해상도가 떨어지고, 얕은 층의 특징 맵은 해상도는 좋지만 의미가 빈약합니다. 이 둘을 합쳐서 작은 객체, 중간 객체, 큰 객체를 모두 잘 잡게 만드는 게 FPN입니다. PAN은 그 반대 방향, 위에서 아래로의 정보 흐름까지 추가합니다.

세 번째는 **헤드**, Head입니다. YOLOv8은 Decoupled Head를 쓰는데, "분리된" 헤드라는 뜻입니다. 바운딩 박스 위치를 예측하는 회귀 헤드와 클래스를 예측하는 분류 헤드를 따로 둡니다. 이게 왜 중요하냐면, 위치 예측과 클래스 예측은 사실 학습 신호가 서로 다른데 이걸 같이 묶어 두면 서로 간섭하거든요. 분리하면 학습이 안정됩니다.

여기까지가 구조의 전반입니다. 이제 모델 크기를 말씀드릴 차례입니다. YOLOv8은 같은 구조 안에서 **다섯 가지 크기**가 제공됩니다.

- **YOLOv8n**, nano, 3.2M 파라미터. 가장 작고 가장 빠릅니다. 엣지 디바이스용입니다.
- **YOLOv8s**, small, 11.2M.
- **YOLOv8m**, medium, 25.9M.
- **YOLOv8l**, large, 43.7M.
- **YOLOv8x**, xlarge, 68.2M. 가장 크고 가장 정확합니다. GPU 서버용이죠.

이 다섯 단계가 있다는 사실 자체가 중요합니다. 같은 모델 패밀리 안에서 정확도-속도 트레이드오프를 슬라이더처럼 조절할 수 있다는 뜻이거든요. 엣지 디바이스에 배포할 거라면 n이나 s를, 서버에서 돌릴 거라면 m이나 l을, 최고 정확도가 필요하면 x를 고르면 됩니다. 실습에서는 n, s, m 세 가지를 비교할 겁니다.

### ③ 경량화의 핵심: Depthwise Separable Convolution (4분)

자, 이제 엣지에서 어떻게 실시간이 가능한지 그 비밀을 들여다보겠습니다. 핵심은 **Convolution 자체를 가볍게 만드는 트릭**입니다. *Practical Deep Learning* 6장에 자세히 나오는 **Depthwise Separable Convolution**이라는 기법이죠. MobileNet에서 처음 대중화됐고, 이후 거의 모든 경량 모델의 기본기가 됐습니다.

먼저 일반 Convolution의 연산량을 한번 보세요. 입력이 H×W×C_in이고 K×K 크기의 필터로 C_out 채널을 만든다고 하면, 연산량은 H × W × K² × C_in × C_out입니다. 224×224 이미지에 3×3 필터, 입력 64채널, 출력 128채널이면 곱셈만 약 3억 7천만 번 일어납니다. 한 레이어에서요. 이게 수십 개 쌓이면 모바일에서는 도저히 실시간이 안 나옵니다.

Depthwise Separable은 이 한 단계 연산을 **두 단계로 쪼개서** 같은 일을 훨씬 적은 계산으로 합니다.

첫 번째 단계가 **Depthwise Convolution**입니다. 입력 채널을 하나하나 따로 처리합니다. 즉 64채널이 들어오면 64개의 독립된 3×3 필터가 각각의 채널에만 작용합니다. PyTorch에서 `groups=in_channels`로 설정하면 됩니다. 이 단계의 역할은 "공간 정보만 추출"하는 거죠. 채널끼리는 섞지 않습니다.

두 번째 단계가 **Pointwise Convolution**입니다. 1×1 크기의 필터로 채널 간 정보를 결합합니다. 1×1이라는 게 핵심인데, 공간 정보는 손대지 않고 채널 차원만 압축하거나 확장하는 역할을 합니다. 여기서 비로소 출력 채널 수가 128로 늘어나는 거죠.

이 두 단계를 합치면, 결과적으로 일반 Convolution과 비슷한 표현력을 가지면서 **연산량은 약 1/8에서 1/9 수준**으로 줄어듭니다. 책에서 인용하는 실험치를 보면, 정확도 손실은 1~2% 정도에 불과합니다. 거의 공짜로 8~9배의 속도 향상을 얻는 셈이죠.

여기에 한 가지 기법을 더 얹으면 엣지 배포가 거의 완성됩니다. **양자화(Quantization)**입니다. *Practical Deep Learning* 6장의 "Quantize the Model" 절에서 다룹니다. 학습된 모델의 가중치를 32비트 부동소수점(FP32) 대신 8비트 정수(INT8)로 저장하는 기법이죠. 모델 크기는 4배 줄고, 모바일에서 추론 속도는 약 2~3배 빨라지고, 정확도 손실은 1% 미만이라고 책에서는 보고합니다. TensorFlow Lite, Core ML, TensorRT 같은 추론 프레임워크가 모두 양자화를 지원합니다.

기억할 키워드는 이 두 개입니다. **Depthwise Separable Convolution**과 **Quantization**. 둘 다 "정확도를 아주 조금 양보하고 속도와 크기를 크게 얻는" 같은 철학을 가진 기법입니다. YOLOv8n이 휴대폰에서도 실시간으로 돌아가는 비결이 정확히 이런 기법들의 누적입니다.

## 1-3. Claude Code 시연 (10분)

자, 이론은 여기까지 하고 직접 만져 볼 시간입니다. Claude Code로 안전모 탐지 데모를 만들어 보겠습니다.

저는 이런 프롬프트로 시작하겠습니다.

```
YOLOv8로 안전모 탐지 데모를 만들어줘.
- ultralytics 라이브러리 사용
- 세 가지 모델 비교: YOLOv8n, YOLOv8s, YOLOv8m
- 동일한 테스트 이미지로 각 모델 추론
- 각 모델의 추론 시간(ms), 탐지 수, bbox 시각화
- 속도 대 mAP 비교 그래프
```

*(시연 진행)*

Claude Code가 코드를 만들어 주는 동안 핵심 흐름을 같이 따라가 보세요.

먼저 라이브러리 import 부분에서 `from ultralytics import YOLO`가 나옵니다. ultralytics가 YOLOv8을 공식적으로 배포하는 라이브러리인데, 가중치 파일까지 자동으로 다운로드해 줍니다. `YOLO('yolov8n.pt')` 한 줄로 사전 학습된 모델이 메모리에 올라오죠. 처음 실행할 때는 인터넷에서 가중치 파일을 받아오니까 몇 초 걸리는데, 두 번째부터는 캐시에서 가져옵니다.

그다음이 **테스트 이미지 준비**입니다. 실제 데모에서는 안전모 미착용자가 포함된 공장 이미지를 쓰지만, 지금은 시간 관계상 무작위 픽셀로 만든 640×640 더미 이미지를 씁니다. 모델이 객체를 찾지는 못해도 추론 자체는 정상적으로 돌아가고, 우리 목적이 속도 비교니까 충분합니다.

세 번째 단계가 가장 중요합니다. **추론 시간 측정**이죠. 코드를 보시면 `_ = model(test_image, verbose=False)`이 먼저 나옵니다. 이게 **웜업**입니다. 첫 번째 추론은 CUDA 커널 초기화, 메모리 할당 같은 비용이 들어가서 실제보다 훨씬 느립니다. 이걸 측정값에 넣으면 결과가 왜곡되니까 한 번 버리는 거죠. *Practical Deep Learning* 6장에서도 벤치마킹할 때 워밍업이 필수라고 강조합니다.

웜업 다음에 10번을 반복해서 측정하고 평균을 냅니다. 한 번만 재면 OS의 컨텍스트 스위치, 디스크 캐시 등 외부 요인 때문에 분산이 큽니다. 여러 번 측정해 평균을 내는 게 정석입니다.

자, 결과가 나왔네요.

YOLOv8n이 약 20밀리초, YOLOv8s가 약 40밀리초, YOLOv8m이 약 80밀리초 근처라고 가정해 봅시다. 정확한 숫자는 머신에 따라 다르고, 지금 화면의 숫자를 같이 보시면 됩니다. 중요한 건 패턴이죠. **모델이 커질수록 추론 시간이 단조 증가**합니다. n에서 s로 갈 때 약 2배, s에서 m으로 갈 때 또 2배. 이게 우리가 이론에서 예측한 그대로의 결과입니다.

여기서 한 가지 직관적이지 않은 사실이 있습니다. **파라미터 수가 두 배가 된다고 추론 시간이 정확히 두 배가 되지는 않는다**는 거죠. 예를 들어 n(3.2M)에서 s(11.2M)로 가면 파라미터는 3.5배 많아지지만 시간은 약 2배만 늘어납니다. 왜 그럴까요?

이유가 세 가지 있습니다. 첫째, GPU의 메모리 대역폭과 연산 처리량 사이의 비율이 모델마다 다릅니다. 작은 모델은 연산보다 메모리 접근이 병목이 되는 경우가 많아서, 연산량 증가가 시간에 그대로 반영되지 않습니다. 둘째, 후처리 시간, 예를 들어 Non-Max Suppression 같은 건 모델 크기와 무관하게 거의 고정값입니다. 작은 모델일수록 이 고정 비용의 비율이 커지죠. 셋째, GPU 워프(warp) 단위로 작업이 묶이니까 모델이 일정 크기를 넘어가야 하드웨어를 꽉 채우게 됩니다. 그 전까지는 모델이 커져도 별로 안 느려져요.

시연 후 질문을 하나 던지겠습니다. **"모델 크기를 반으로 줄이면 속도가 정확히 두 배가 되는 건 아닌데, 왜 그럴까?"** 방금 설명한 세 가지가 다 답이 될 수 있습니다. 이건 강의 후반에 다시 한번 짚고 가겠습니다.

## 2-2. 이론 (13분)

### ① 비디오 이해의 접근법: 왜 골격 기반인가 (3분)

영상에서 행동을 인식하는 방법은 크게 두 갈래로 나뉩니다. *Practical ML for Computer Vision* 11장에서 양쪽을 비교해서 설명하는데, 그 비교를 그대로 옮기면서 실무 관점을 더해 보겠습니다.

첫 번째 갈래가 **픽셀 기반 접근**입니다. 3D CNN이나 Two-Stream 네트워크가 대표적이죠. 영상 프레임을 그대로 입력으로 받아서 시공간 패턴을 학습합니다. 장점은 배경 정보까지 활용하니까 정확도가 높을 수 있다는 점입니다. 단점은 무겁다는 거예요. 연산량이 워낙 커서 엣지 디바이스에서는 거의 불가능합니다. 그리고 픽셀을 직접 다루니까 사생활 침해 우려도 크고, 조명이나 카메라 각도 변화에도 취약합니다.

두 번째 갈래가 **골격 기반 접근**, Skeleton/Keypoint 방식입니다. 영상에서 사람의 관절 좌표만 뽑아내고, 그 좌표 시퀀스를 분류합니다. 장점이 많습니다. 일단 가볍습니다. 관절 33개 좌표만 다루니까 데이터량이 픽셀의 수백분의 일이죠. **엣지 디바이스에서도 실시간 처리가 가능합니다.** 또 좌표만 쓰니까 개인정보 보호 측면에서 유리합니다. 누구인지 식별할 수 없거든요. 그리고 좌표는 조명이나 배경 변화에 강건합니다. 어두운 곳에서도, 배경이 바뀌어도, 사람의 관절 위치는 비슷하게 추출됩니다.

단점도 분명히 있습니다. **가림, 즉 occlusion에 취약**합니다. 작업자가 기계 뒤에 가려지면 관절이 안 보이고, 그러면 행동 인식 정확도가 떨어집니다. 또 미세한 동작, 예를 들어 손가락 움직임 같은 건 33개 키포인트로는 표현이 안 됩니다.

자, 우리가 만들 시스템의 요구 사항을 다시 보세요. 24시간 운영, 엣지 디바이스 배포, 개인정보 보호. 이 세 가지가 동시에 요구된다면 **답은 골격 기반 접근**입니다. 책의 결론도 같습니다. "Engineering 관점에서, 골격 기반이 현실적인 선택"이죠.

### ② MediaPipe로 골격 추출 (5분)

골격 기반 접근의 첫 번째 단계는 영상에서 키포인트를 뽑는 것입니다. 여기서 우리가 쓸 도구가 **MediaPipe Pose**입니다. 구글이 만든 오픈소스 라이브러리고, 산업 현장에서 가장 많이 쓰이는 포즈 추정 도구 중 하나입니다.

MediaPipe Pose는 이미지나 영상 프레임에서 **33개의 키포인트** 좌표를 실시간으로 추출합니다. 33개를 어떻게 배치했는지 잠깐 보여드릴게요.

```
0번 코
1~8번 얼굴
11·12번 어깨
13·14번 팔꿈치, 15·16번 손목
17~22번 손 (각 손에 3개씩)
23·24번 엉덩이
25·26번 무릎, 27·28번 발목
29~32번 발 (각 발에 2개씩)
```

각 키포인트는 세 가지 값을 가집니다. **x, y, visibility**. x와 y는 이미지를 0과 1 사이로 정규화한 좌표값이고, visibility는 이 키포인트가 얼마나 잘 보이는지를 0과 1 사이로 나타내는 값입니다. 가려지면 낮아지고, 또렷이 보이면 높아집니다.

33개 × 3값 = **한 프레임이 99차원 벡터**로 표현됩니다. 이 99차원 벡터가 앞으로 우리가 다룰 데이터의 단위입니다.

코드로 보면 이렇습니다.

```python
import mediapipe as mp, cv2, numpy as np

mp_pose = mp.solutions.pose

def extract_keypoints(frame):
    with mp_pose.Pose(
        min_detection_confidence=0.5,
        min_tracking_confidence=0.5
    ) as pose:
        rgb = cv2.cvtColor(frame, cv2.COLOR_BGR2RGB)
        results = pose.process(rgb)
        if results.pose_landmarks:
            kp = np.array([
                [lm.x, lm.y, lm.visibility]
                for lm in results.pose_landmarks.landmark
            ])
            return kp
        return np.zeros((33, 3))
```

여기서 짚을 점이 몇 가지입니다.

`min_detection_confidence=0.5`는 "50% 미만의 확신은 무시한다"는 뜻입니다. 너무 낮추면 잡음이 들어오고, 너무 높이면 진짜 사람을 놓칩니다. 0.5가 기본값이고 대부분 잘 동작합니다.

`cv2.cvtColor(frame, cv2.COLOR_BGR2RGB)`는 색상 채널 순서 변환입니다. OpenCV는 BGR로 읽어오는데 MediaPipe는 RGB를 원합니다. 이걸 빼먹으면 결과가 미묘하게 이상해지죠. 디버깅하기 어려운 함정 중 하나입니다.

키포인트가 검출되지 않으면 33×3 영행렬을 반환합니다. 시퀀스 입력 차원을 일정하게 유지하기 위함이죠.

그리고 가장 좋은 점. **MediaPipe는 CPU만으로도 초당 30프레임이 가능합니다.** GPU가 필요 없으니까 Raspberry Pi 같은 엣지 디바이스에서도 충분히 돌아갑니다. 이게 우리가 골격 기반 접근을 택한 이유의 핵심 근거 중 하나입니다.

### ③ 키포인트 시퀀스를 LSTM으로 분류 (5분)

자, 한 프레임의 키포인트는 뽑았습니다. 그런데 행동은 **한 프레임으로 판단할 수 없죠**. "허리를 굽혔다"는 상태고, "허리를 굽히는 동작 중이다"는 시간에 따른 변화입니다. 따라서 행동 인식은 본질적으로 **시퀀스 분류 문제**입니다.

여기서 Session 2에서 배운 LSTM이 다시 등장합니다. Session 2에서는 센서 시계열을 LSTM에 넣어서 RUL을 예측했죠. 똑같은 원리로, 이번에는 키포인트 시계열을 LSTM에 넣어서 행동을 분류합니다.

전체 파이프라인은 이렇게 됩니다.

```
비디오 프레임 시퀀스
      ↓ MediaPipe
키포인트 시퀀스 [30 × 99]
      ↓ LSTM
행동 분류 결과 ["정상 작업" / "위험 행동A" / "위험 행동B"]
```

30이라는 숫자는 어디서 왔냐. **슬라이딩 윈도우 크기**입니다. 30FPS 영상에서 30프레임이면 1초 분량이죠. 1초 정도면 대부분의 작업 동작 단위를 포착할 수 있습니다.

윈도우는 보통 겹치게 잡습니다. `window_size=30, step=15`이면 30프레임 단위 윈도우를 만들고, 15프레임씩 이동하면서 다음 윈도우를 만듭니다. 절반이 겹치는 거죠. 이렇게 하면 1초마다가 아니라 0.5초마다 결과가 갱신됩니다. 반응 속도가 두 배 빨라지죠.

코드는 이렇습니다.

```python
def create_sequences(keypoint_data, labels, window_size=30, step=15):
    X, y = [], []
    for i in range(0, len(keypoint_data) - window_size, step):
        X.append(keypoint_data[i:i+window_size].reshape(window_size, -1))
        y.append(labels[i+window_size])
    return np.array(X), np.array(y)
```

이렇게 만든 시퀀스를 LSTM 모델에 넣습니다.

```python
model = tf.keras.Sequential([
    tf.keras.layers.LSTM(64, input_shape=(30, 99), return_sequences=True),
    tf.keras.layers.Dropout(0.3),
    tf.keras.layers.LSTM(32),
    tf.keras.layers.Dense(32, activation='relu'),
    tf.keras.layers.Dense(3, activation='softmax')
])
model.compile(optimizer='adam',
              loss='sparse_categorical_crossentropy',
              metrics=['accuracy'])
```

구조를 한 줄씩 짚어 볼게요.

`LSTM(64)`는 첫 번째 LSTM 층입니다. 64는 은닉 상태의 차원입니다. `input_shape=(30, 99)`는 "30 타임스텝, 99 차원"이라는 뜻이고요. `return_sequences=True`는 다음 LSTM 층이 또 오니까 시퀀스 전체를 다음 층에 넘기겠다는 표시입니다.

`Dropout(0.3)`은 과적합을 방지하기 위해 학습 중에 30%의 뉴런을 무작위로 꺼두는 정규화 기법입니다.

`LSTM(32)`는 두 번째 LSTM 층입니다. 여기서는 `return_sequences`가 없으니까 마지막 타임스텝의 출력만 반환합니다.

`Dense(32, relu)`는 일반 완전연결층이고, 마지막 `Dense(3, softmax)`가 3개 클래스에 대한 확률을 출력합니다.

손실 함수가 `sparse_categorical_crossentropy`인데, 이건 라벨이 원-핫이 아니라 정수 0, 1, 2로 들어올 때 쓰는 형태입니다. 메모리도 절약되고 코드도 단순해집니다.

다시 한번 강조할게요. **Session 2의 LSTM과 본질적으로 같은 구조**입니다. 입력의 의미만 센서 시계열에서 키포인트 시계열로 바뀐 거죠. 한 번 익혀 둔 모듈은 도메인을 옮겨서 재활용할 수 있다, 이게 딥러닝 도구함의 매력입니다.

## 2-3. Claude Code 시연 (10분)

자, 이제 직접 만들어 봅니다. 이번 시연의 포인트는 **MediaPipe 출력을 LSTM 입력으로 어떻게 변환하는가**입니다. 두 도구 사이의 데이터 흐름을 손으로 잡아 보는 게 목표입니다.

프롬프트는 이렇게 시작합니다.

```
작업자 위험 행동 인식 모델을 만들어줘.

- window_size=30 프레임으로 슬라이딩 윈도우 생성
- LSTM(64) → Dropout(0.3) → LSTM(32) → Dense(3, softmax)
- 행동 클래스: ["정상작업", "허리위험각도", "고소작업위험"]
- 합성 데이터로 학습
```

*(시연 진행)*

코드가 만들어지는 동안 단계별 흐름을 짚어 보겠습니다.

먼저 **합성 데이터를 만드는 부분**부터 보세요. 실제 산업 현장 데이터를 모으는 건 시간이 오래 걸리니까, 데모 단계에서는 키포인트 패턴을 인위적으로 만들어 학습시킵니다. 정상 작업은 약간의 무작위 잡음만, 허리 위험은 엉덩이 키포인트 좌표가 윈도우 안에서 점진적으로 변하는 패턴, 고소 작업 위험은 y좌표가 전반적으로 위쪽인 패턴, 이런 식이죠. 모델이 이 차이를 학습할 수 있는지가 검증 포인트입니다.

다음으로 **윈도우 생성**. `create_sequences` 함수가 키포인트 시퀀스를 30프레임짜리 윈도우로 잘라줍니다. 출력 shape은 (배치, 30, 99)가 되죠.

학습은 빠르게 수렴할 겁니다. 패턴이 명확하니까요. 실제 데이터에서는 이렇게 깨끗하지는 않습니다. 그래도 모델 구조와 입출력 형태를 먼저 잡아 두는 게 중요합니다.

자, 결과가 나왔네요. 정확도가 90% 후반대까지 올라갔을 겁니다. 합성 데이터 기준이니까 너무 좋아하지는 마시고, 패턴이 잘 잡힌다는 증거 정도로 받아들이세요.

시연 후 질문을 던지겠습니다. **"카메라가 정면이 아니라 측면에서 찍히면 키포인트가 어떻게 달라지고, 어떻게 대응해야 할까?"**

생각해 보세요. 측면에서는 한쪽 팔, 한쪽 다리가 다른 쪽에 가려집니다. visibility 값이 낮아지죠. 어떻게 대응할까요?

세 가지 방법이 있습니다.

첫째, **visibility 가중치**. LSTM 입력에서 visibility가 낮은 키포인트의 영향을 줄이는 거죠. 마스킹이나 visibility를 직접 가중치로 사용하는 방법이 있습니다.

둘째, **다중 카메라 융합**. 정면과 측면 두 대를 두고 더 잘 보이는 쪽의 키포인트를 우선 사용합니다. 비싸지만 효과적입니다.

셋째, **학습 데이터에 다양한 각도 포함**. 측면, 후면, 비스듬한 각도 영상을 학습 데이터에 골고루 섞어 두면 모델이 각도 변화에 자연스럽게 강건해집니다. 가장 본질적인 해결책이죠.

## 3-1. 파이프라인 설계 + 데이터 준비 (10분)

자, 마지막 섹션입니다. 지금까지 우리는 두 개의 독립적인 기능을 만들었습니다. 섹션 1의 YOLO는 객체를 잡고, 섹션 2의 MediaPipe+LSTM은 행동을 잡습니다. 둘 다 잘 작동하지만, 따로따로 돌리면 그저 두 개의 데모일 뿐이죠. 이걸 **하나의 CCTV 안전 위반 탐지 파이프라인**으로 엮는 게 섹션 3의 목표입니다.

전체 구조를 먼저 그려 보겠습니다.

```
CCTV 영상 입력 (30 FPS)
      ↓ 프레임 추출
단일 프레임
      ↓
┌─────────────────────────────────────────┐
│  병렬 처리                                  │
│                                            │
│  [YOLO]            → 안전모 탐지              │
│    bbox 신뢰도 > 0.5? → 미착용 위반 판정      │
│                                            │
│  [MediaPipe + LSTM] → 행동 인식              │
│    위험 행동 확률 > 0.7? → 행동 위반 판정     │
└─────────────────────────────────────────┘
      ↓ 위반 판정 로직
  위반 조건 충족?
      ↓ Yes                ↓ No
  경보 발생                정상 처리
  프레임 저장              계속 모니터링
```

이 구조에서 **병렬 처리**가 핵심입니다. 왜 병렬이어야 하는가? 두 가지 모델이 **자연스럽게 서로 다른 호흡으로 동작하기 때문**입니다.

YOLO는 단일 프레임 입력입니다. 매 프레임마다 추론할 수 있고, 한 프레임의 결과만으로 의사 결정이 가능하죠. "안전모 있다/없다"는 한 프레임으로 판단 가능합니다.

반면 LSTM은 30프레임 시퀀스가 모여야 비로소 추론이 가능합니다. 첫 30프레임은 결과가 안 나오고, 그 이후부터 슬라이딩 윈도우 step 단위로 결과가 갱신되죠. 호흡이 느립니다.

이 둘을 직렬로 묶으면, 즉 매 프레임마다 "YOLO 끝나면 LSTM 한다" 식으로 짜면 LSTM이 결과를 못 내고 있을 때 YOLO 결과도 같이 멈춰버리는 사태가 납니다. 그래서 **각 스트림이 자기 속도로 돌아가게 하고, 결과를 OR 조건으로 합쳐서 위반 여부를 판정**합니다.

데이터 준비 측면에서 몇 가지 짚을 점이 있습니다. *Practical ML for Computer Vision* 5장에서는 데이터셋을 만들 때의 주의점을 자세히 다루는데, 그 중 우리 시나리오에 직접 관련 있는 것들만 정리해 보겠습니다.

첫째, **카메라 위치와 각도**. 안전모 탐지에는 머리가 잘 보이는 각도가, 행동 인식에는 전신이 잘 보이는 각도가 필요합니다. 가능하면 한 대의 카메라로 둘 다 잡을 수 있게 약간 높은 위치에 설치합니다.

둘째, **클래스 라벨링 규칙의 명확성**. "안전모 미착용"이 어떤 상태인지 합의가 없으면 라벨이 흔들립니다. 턱끈을 안 매면 미착용인지, 머리에 얹어만 놔도 착용인지 같은 경계 케이스를 미리 정의하세요. 책에서는 라벨링 가이드라인 문서를 데이터셋과 함께 관리하라고 권합니다.

셋째, **데이터 불균형**. 위반 상황은 정상 상황보다 훨씬 드뭅니다. 그대로 학습하면 모델이 "정상"만 외워도 정확도가 90%대가 나옵니다. 그래서 위반 클래스를 의도적으로 오버샘플링하거나, 클래스 가중치를 손실 함수에서 조정합니다.

오늘은 시간 관계상 미리 준비된 합성 데이터와 사전 학습 모델을 그대로 쓸 겁니다. 하지만 실무에서는 이 데이터 준비 단계가 전체 프로젝트 공수의 절반 이상을 차지한다는 걸 기억해 두세요.

## 3-2. 통합 탐지 + 알림 트리거 (15분)

이제 통합 탐지 함수를 만들어 보겠습니다. 핵심은 **하나의 함수에 YOLO와 MediaPipe+LSTM을 모두 묶고, 위반 여부와 시각화된 프레임을 반환**하는 것입니다.

함수 시그니처를 먼저 봅시다.

```python
def detect_violation(frame, yolo_model, action_model, keypoint_buffer,
                     helmet_threshold=0.5, action_threshold=0.7):
    violations = []
    annotated = frame.copy()
    ...
    return is_violation, violations, annotated
```

입력은 현재 프레임, YOLO 모델, 행동 분류 모델, 그리고 키포인트 버퍼입니다. 키포인트 버퍼가 중요한 인자인데, 이게 바로 LSTM이 시퀀스를 모으는 메모리 역할을 합니다. `collections.deque(maxlen=30)`으로 만들어서 항상 최근 30프레임만 유지하면 깔끔하죠.

신뢰도 임계값이 두 개 있습니다. **`helmet_threshold=0.5`**는 YOLO 바운딩 박스가 안전모로 인정되려면 최소 50% 신뢰도가 필요하다는 뜻이고, **`action_threshold=0.7`**은 LSTM의 행동 클래스 확률이 70% 이상이어야 위반으로 인정한다는 뜻입니다. 안전 모니터링은 보수적으로 잡아도 되니까 행동 쪽은 좀 높게 설정합니다.

### YOLO 스트림

```python
yolo_results = yolo_model(frame, verbose=False)
for box in yolo_results[0].boxes:
    cls = int(box.cls)
    conf = float(box.conf)
    if cls == 0 and conf > helmet_threshold:
        x1, y1, x2, y2 = map(int, box.xyxy[0])
        cv2.rectangle(annotated, (x1, y1), (x2, y2), (0, 0, 255), 2)
        cv2.putText(annotated, f'NO HELMET {conf:.2f}',
                   (x1, y1-10), cv2.FONT_HERSHEY_SIMPLEX, 0.6, (0, 0, 255), 2)
        violations.append('helmet_violation')
```

YOLO 결과는 `boxes` 객체 안에 들어 있습니다. 각 박스의 클래스, 신뢰도, 좌표(x1, y1, x2, y2)를 꺼내서 임계값을 통과한 것만 처리합니다. `cls == 0`은 우리가 학습할 때 "안전모 미착용" 클래스에 0번을 부여했다고 가정한 거죠. 실제 라벨 순서에 맞춰 바꿔 주세요. `cv2.rectangle`로 빨간 박스를 그리고, `cv2.putText`로 신뢰도까지 표시합니다.

### MediaPipe + LSTM 스트림

```python
keypoints = extract_keypoints(frame)
keypoint_buffer.append(keypoints.flatten())

if len(keypoint_buffer) >= 30:
    sequence = np.array(list(keypoint_buffer)[-30:])
    pred = action_model.predict(sequence[np.newaxis, ...], verbose=0)[0]
    action_cls = np.argmax(pred)
    action_conf = pred[action_cls]

    action_names = ['정상작업', '허리위험', '고소위험']
    if action_cls != 0 and action_conf > action_threshold:
        label = f'{action_names[action_cls]} {action_conf:.2f}'
        cv2.putText(annotated, label, (10, 30),
                   cv2.FONT_HERSHEY_SIMPLEX, 1, (0, 0, 255), 2)
        violations.append(f'action_{action_names[action_cls]}')
```

이 블록을 한 줄씩 보세요. 키포인트를 뽑고, flatten해서 99차원 벡터로 만든 다음, 버퍼에 추가합니다. 버퍼가 30개에 도달했을 때만 LSTM 추론을 호출합니다. 그 전에는 결과를 못 내니까 그냥 패스하는 거죠.

LSTM 예측에서 `action_cls != 0`은 "정상작업이 아닌 경우만"이라는 조건입니다. 0이 정상작업 인덱스라고 가정한 거죠. 위반 행동(1번 허리 위험, 2번 고소 위험)이면서 신뢰도가 threshold를 넘으면 빨간 글씨로 라벨링하고 위반 목록에 추가합니다.

### 위반 이벤트 로깅

탐지된 위반은 그냥 화면에 띄우고 끝나면 안 됩니다. **CSV 파일로 기록**해서 사후 분석을 가능하게 해야 합니다. 어느 시간대에 어떤 위반이 많이 일어났는지, 특정 작업자나 특정 라인에서 패턴이 있는지 같은 걸 데이터로 보려면 로깅이 필수입니다.

```python
import csv
from datetime import datetime

def log_violation(frame_idx, fps, violation_types, log_path='violations.csv'):
    timestamp = frame_idx / fps
    with open(log_path, 'a', newline='') as f:
        writer = csv.writer(f)
        writer.writerow([
            datetime.now().strftime('%Y-%m-%d %H:%M:%S'),
            f'{timestamp:.1f}s',
            ', '.join(violation_types)
        ])
```

타임스탬프는 두 가지를 기록합니다. 실제 벽시계 시간(언제 일어난 일인지)과 영상 내 상대 시간(영상의 몇 초 지점인지). 둘 다 있으면 영상에서 해당 장면을 다시 찾아보기 쉽습니다.

### 오탐 감소: 연속 프레임 규칙

여기서 매우 중요한 실무 이슈가 하나 있습니다. **오탐**, false positive를 어떻게 줄일 것인가.

지금까지 만든 함수를 그대로 돌리면, YOLO가 한 프레임에서 살짝 흔들려서 잠깐 안전모를 못 보거나, MediaPipe가 한 프레임 키포인트를 미세하게 잘못 잡으면 바로 위반 알림이 발생합니다. 30분 가동에 100번씩 거짓 알림이 울리면 현장에서 시스템을 꺼버립니다. 안전 시스템의 가장 큰 적은 **알람 피로**, alarm fatigue거든요.

해결책은 단순하고 강력합니다. **N개 연속 프레임 규칙**입니다. "한 프레임에서만 위반이 보이면 무시하고, N개 프레임 연속으로 위반이 지속될 때만 알림을 보낸다." 30FPS에서 N=15면 0.5초 이상 지속되는 위반만 잡는 거죠.

```python
class ViolationDetector:
    def __init__(self, min_consecutive=15):
        self.min_consecutive = min_consecutive
        self.consecutive_count = {}

    def update(self, violation_types):
        active = set(violation_types)
        triggered = []
        for v in active:
            self.consecutive_count[v] = self.consecutive_count.get(v, 0) + 1
            if self.consecutive_count[v] == self.min_consecutive:
                triggered.append(v)
        # 더 이상 보이지 않는 위반은 카운트 리셋
        for v in list(self.consecutive_count.keys()):
            if v not in active:
                self.consecutive_count[v] = 0
        return triggered
```

이 클래스를 detect_violation의 결과와 함께 사용하면, 일시적 오탐이 제거됩니다. *Practical ML for Computer Vision* 책에서도 다단계 탐지 파이프라인의 조건 결합 전략을 다룰 때 이런 시간적 평활화(temporal smoothing)를 표준 기법으로 소개합니다.

여기까지가 통합 탐지 함수의 골격입니다. 코드는 길지만 흐름은 단순하다는 걸 느끼셨으면 합니다. **두 개의 독립적인 스트림, 각각의 임계값, OR 결합, 그리고 시간적 평활화.** 이 네 가지가 안전 모니터링 파이프라인의 본질이에요.

기본을 끝낸 분은 중급으로 가세요. **임계값이 결과에 어떤 영향을 주는지**를 데이터로 확인하는 실험입니다.

1. `helmet_threshold`를 0.3, 0.5, 0.7로 바꿔가며 탐지 이벤트 수를 기록합니다.
2. 각 임계값에서 잘못 잡힌(또는 놓친) 대표 프레임을 추출해서 비교합니다.

제출물: **임계값별 탐지 이벤트 수 표 + "제조 현장에서 어떤 임계값을 선택할 것인가"에 대한 짧은 한 문단의 의견.**

여기서 한 가지 힌트. **안전 모니터링에서는 Recall이 Precision보다 우선**입니다. 즉, "한 번이라도 놓치면 안 된다"가 "가짜로 잡힌 게 좀 있다"보다 비쌉니다. 임계값을 낮춰서 Recall을 높이고, 오탐은 앞서 만든 연속 프레임 규칙으로 잡는다, 이게 일반적인 운영 전략입니다. 그러나 너무 낮추면 알람 피로가 폭발하니까 균형점을 찾아야 하죠.

### 심화 — 위반 이벤트 로그 분석 대시보드

3. **누적 위반 시계열**. x축이 시간, y축이 누적 위반 수인 시계열 그래프입니다. 기울기가 가파른 구간이 위반 다발 시간대죠.

`matplotlib`이나 `plotly`로 충분합니다. 더 욕심내신다면 `streamlit` 한 페이지로 묶어서 인터랙티브 대시보드를 만들어 보셔도 좋습니다.
