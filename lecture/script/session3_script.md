# 섹션 1 — Object Detection: YOLO가 어떻게 그렇게 빠른가 (45분)

## 1-1. 문제 제기 (3분)

공장에 CCTV가 설치되어 있다고 상상해 보세요. 실시간으로 안전을 모니터링하려면 어떻게 해야 할까요?

가장 단순한 방법은 모든 영상을 GPU 서버로 보내서 분석하는 겁니다. 하지만 문제가 있습니다. 지연 시간이 0.5~2초 발생합니다. 24시간 영상을 보내려면 막대한 네트워크 비용이 듭니다. 그리고 공장 내부 영상을 외부로 전송하는 건 보안상 문제가 될 수 있습니다.

해결책은 **엣지 컴퓨팅**입니다. Jetson Nano나 Raspberry Pi를 카메라 옆에 배치해서 즉시 분석하는 거죠. 지연 시간 없고, 네트워크도 필요 없고, 보안도 유지됩니다.

그런데 핵심 질문이 생깁니다. **GPU 없는 작은 컴퓨터에서 실시간 탐지가 가능한가?** 정확도와 속도의 트레이드오프를 어떻게 해결할 것인가?

---

## 1-2. 이론 (13분)

### ① Object Detection의 발전: 왜 YOLO인가 (4분)

*Practical Machine Learning for Computer Vision* 4장과 5장 내용입니다.

Object Detection은 크게 두 가지 방식으로 발전했습니다.

**2단계 탐지기**, R-CNN 계열은 먼저 영역을 제안하고, 그 다음 분류합니다. 정확도는 높지만 느립니다. 초당 2~10프레임이죠. 실시간으로는 사용할 수 없습니다.

**1단계 탐지기**, YOLO 계열은 단일 추론으로 위치와 클래스를 동시에 예측합니다. 정확도는 약간 낮지만 빠릅니다. 초당 30~100프레임 이상이죠.

제조 현장 실시간 안전 모니터링에서는 YOLO가 **사실상의 표준**입니다.

### ② YOLOv8 구조 직관 (5분)

YOLOv8의 구조는 세 부분으로 나뉩니다.

**백본**, CSPDarknet은 이미지에서 특징을 추출합니다. 텍스처, 모서리, 형태 같은 거죠.

**넥**, FPN + PAN은 다중 스케일 특징을 융합합니다. 작은 객체, 중간 객체, 큰 객체를 모두 탐지할 수 있게 하는 거죠.

**헤드**, Decoupled Head는 바운딩 박스 좌표와 클래스를 동시에 예측합니다.

YOLOv8은 모델 크기에 따라 다섯 가지 버전이 있습니다. Nano가 3.2M 파라미터로 엣지 디바이스용, Small가 11.2M, Medium이 25.9M, Large가 43.7M, XLarge가 68.2M으로 서버용입니다.

### ③ 경량화의 핵심: Depthwise Separable Convolution (4분)

*Practical Deep Learning for Cloud, Mobile, and Edge* 6장 내용입니다.

엣지 디바이스에서 실시간 탐지를 하려면 모델을 가볍게 만들어야 합니다. 핵심 기술이 **Depthwise Separable Convolution**입니다.

일반 Convolution은 입력 채널과 출력 채널을 모두 고려해서 연산합니다. 64채널 입력을 128채널로 변환하려면 64×128개의 필터가 필요하죠.

Depthwise Separable Convolution은 이걸 두 단계로 나눕니다.

첫째, **Depthwise Convolution**. 각 채널별로 따로따로 합성곱을 수행합니다. `groups=in_channels`로 설정하면 됩니다.

둘째, **Pointwise Convolution**. 1×1 합성곱으로 채널 간 정보를 결합합니다.

결과적으로 연산량이 약 **1/8에서 1/9**로 줄어듭니다. 정확도 손실은 1~2%에 불과합니다. 엣지 배포를 위한 핵심 기술입니다.

# 섹션 2 — Action Recognition: 사람의 행동을 숫자로 읽는다 (45분)

> 참고 교재: *Practical Machine Learning for Computer Vision* Ch.5
> 소요 시간: 문제 제기 3분 + 이론 13분 + 시연 10분 + 실습 19분

---

## 2-1. 문제 제기 (3분)

섹션 1에서는 "무엇이 있는가", 객체 탐지를 다뤘습니다. 이번에는 한 단계 더 나아갑니다.

**"그 사람이 무엇을 하고 있는가"**를 인식할 수 있는가?

제조 현장에서 위험 행동의 예를 들어볼까요. 2인 1조 규정인데 혼자 무거운 물건을 이동하는 경우. 안전 난간 없이 고소 작업을 하는 경우. 보호구를 미착용하고 위험 구역에 진입하는 경우. 위험한 수준으로 허리를 굽히는 경우.

이런 행동을 실시간으로 감지할 수 있어야 합니다.

---

## 2-2. 이론 (13분)

### ① 비디오 이해의 접근법: 왜 골격 기반인가 (3분)

행동 인식에는 크게 두 가지 접근법이 있습니다.

**픽셀 기반 접근**, 3D CNN이나 Two-Stream 방식은 정확도가 높지만 무겁습니다. 또 개인정보 침해 우려가 있고, 조명이나 각도에 취약합니다.

**골격 기반 접근**, Skeleton/Keypoint 방식은 가볍습니다. 엣지에서도 실행 가능하고, 좌표만 사용하니까 개인정보 보호도 되고, 조명이나 배경 변화에도 강건합니다. 단점은 가림, occlusion에 취약하다는 겁니다.

제조 현장은 24시간 운영, 엣지 배포, 개인정보 보호가 요구되기 때문에 **골격 기반 접근이 실용적인 선택**입니다.

### ② MediaPipe로 골격 추출 (5분)

MediaPipe Pose는 이미지나 영상에서 **33개의 키포인트**를 실시간으로 추출합니다. 코, 얼굴 8개, 어깨 2개, 팔꿈치 2개, 손목 2개, 손 6개, 엉덩이 2개, 무릎 2개, 발목 2개, 발 4개입니다.

각 키포인트는 x, y, visibility 세 가지 값을 가집니다. 33개 × 3 = 99차원 벡터가 한 프레임의 표현이 되는 거죠.

가장 좋은 건 **CPU만으로도 초당 30프레임**이 가능하다는 겁니다. GPU가 필요 없으니까 엣지 배포에 매우 실용적입니다.

### ③ 키포인트 시퀀스를 LSTM으로 분류 (5분)

행동은 단일 프레임이 아니라 **시간에 따른 시퀀스**로 판단해야 합니다.

파이프라인은 이렇습니다. 비디오 프레임 → MediaPipe로 키포인트 추출 → 키포인트 시퀀스 [30×99] → LSTM → 행동 클래스. 정상 작업, 위험 행동 A, 위험 행동 B로 분류합니다.

슬라이딩 윈도우를 사용합니다. `window_size=30`, `step=15`. 30프레임 단위로 시퀀스를 만들고 15프레임씩 이동합니다.

LSTM 모델은: `input_shape=(30, 99)`의 `LSTM(64)` → `Dropout(0.3)` → `LSTM(32)` → `Dense(32, relu)` → `Dense(3, softmax)`. 손실함수는 `sparse_categorical_crossentropy`입니다.

Session 2에서 배운 LSTM이 여기서 다시 등장합니다. RUL 예측에서는 센서 시계열을 입력으로 썼고, 여기서는 키포인트 시계열을 입력으로 쓰는 거죠. 같은 원리입니다.

## 3-1. 파이프라인 설계 + 데이터 준비 (10분)

마지막 섹션입니다. 섹션 1의 YOLO와 섹션 2의 MediaPipe+LSTM을 결합해서 **CCTV 안전 위반 탐지 파이프라인**을 만들겠습니다.

파이프라인 구조는 이렇습니다.

CCTV 영상이 30FPS로 들어옵니다. 프레임을 추출합니다. 그다음 **병렬 처리**가 핵심입니다.

**YOLO 스트림**: confidence > 0.5일 때 안전모 미착용 위반을 탐지합니다.
**MediaPipe+LSTM 스트림**: probability > 0.7일 때 위험 행동을 인식합니다.

두 스트림 중 하나라도 위반을 감지하면 경고를 발생시킵니다.

이 병렬 아키텍처가 중요한 이유는 객체 탐지와 행동 인식이 서로 다른 속도로 실행되기 때문입니다. YOLO는 프레임마다 실행 가능하지만 LSTM은 30프레임 시퀀스가 모여야 실행할 수 있죠.

---

## 3-2. 통합 탐지 + 알림 트리거 (15분)

### 위반 탐지 함수

`detect_violation` 함수를 만듭니다. 입력으로 프레임을 받아서, YOLO로 안전모 탐지를 하고, MediaPipe로 키포인트를 추출하고, LSTM으로 행동을 분류합니다.

YOLO 결과는 바운딩 박스, 신뢰도, 시각적 주석으로 나타냅니다. MediaPipe+LSTM 결과는 키포인트 버퍼 업데이트, 시퀀스 예측, 작업 이름 레이블로 나타냅니다.

### 위반 이벤트 로깅

모든 위반 이벤트는 CSV로 기록합니다. 타임스탬프, 위반 유형, 프레임 번호가 포함됩니다.

### 오탐 감소

실제 배포에서 중요한 문제가 오탐입니다. 단일 프레임에서 한 번 위반이 감지되었다고 바로 경고를 보내면 안 됩니다.

해결책은 **N개 연속 프레임 규칙**입니다. N개의 연속 프레임에서 위반이 지속될 때만 경고를 보내는 거죠. `ViolationDetector` 클래스로 구현합니다.

## 클로징 (3분)

오늘 Session 3에서 세 가지를 배웠습니다.
