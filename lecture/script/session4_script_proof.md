## 1-1. 문제 제기 (3분)

상황을 하나 그려 보겠습니다.

여러분이 어느 제조 공장의 AI 엔지니어라고 해 볼게요. 라인에서 나오는 제품의 표면 결함을 자동으로 탐지하는 모델을 만들어야 합니다. Session 3에서 YOLO를 배웠으니까 도구는 있죠. 그런데 실제로 데이터를 모아 보면 현실은 이렇습니다.

정상 제품 이미지는 **수만 장**이 쉽게 모입니다. 라인이 멀쩡히 돌면 정상 제품이 끊임없이 나오니까요. **스크래치 결함**은 그래도 좀 모입니다. 수백 장 정도. **버블(기포) 결함**은 수십 장. **균열(크랙) 결함**은 아예 한 자릿수, 그러니까 학습이 불가능한 수준입니다.

이게 제조 데이터의 진짜 현실입니다. **결함은 애초에 희귀합니다.** 자주 일어나면 그건 결함이 아니라 공정 문제죠.

Session 1에서는 이런 데이터 불균형을 SMOTE로 해결했습니다. 소수 클래스 샘플의 특징 벡터를 보간해서 새 샘플을 만드는 방법이었죠. 그런데 **이미지 데이터에 SMOTE를 그대로 적용하면** 어떻게 될까요?

두 결함 이미지의 픽셀을 평균 내면 의미 없는 흐릿한 혼합 이미지가 나옵니다. 스크래치 반에 버블 반이 섞인 이미지는 그 어떤 결함도 아닙니다. 그저 어수선한 픽셀 덩어리죠. 모델 입장에서는 노이즈만 추가된 셈입니다.

우리에게 필요한 건 **"진짜처럼 보이는, 그러면서도 새로운"** 결함 이미지입니다. 단순한 보간이 아니라, 새로운 결함의 패턴을 가진 이미지를 합성해 줄 수 있는 도구가 필요한 거죠.

오늘 섹션 1의 핵심 질문은 단 하나입니다. **"세상에 존재하지 않는 결함 이미지를, 진짜처럼 만들어 낼 수 있는가?"** 그 답이 바로 Diffusion Model입니다.

## 1-2. 이론 (13분)

### STEP 01 — Diffusion의 직관: 노이즈 추가 → 노이즈 제거 (5분)

*Hands-On Generative AI* 4장에서 Diffusion 모델을 설명하는 한 줄 문장이 있습니다. **"모델은 노이즈 제거를 학습한다."** 이 한 줄에 본질이 다 들어 있습니다.

먼저 **순방향 과정**, Forward Process를 봅시다.

원본 이미지 x₀에서 시작합니다. 여기에 가우시안 노이즈를 조금 추가합니다. 결과는 x₁, 약간 흐려졌죠. 또 노이즈를 추가합니다. x₂, 좀 더 흐려졌습니다. 이걸 **T번 반복**합니다. 보통 T는 1000이에요. 결국 마지막 단계 x_T는 원본의 흔적이 사라진 **순수 가우시안 노이즈**가 됩니다.

이 과정에는 학습할 게 없습니다. 단순한 수학 공식이죠. q(x_t | x_{t-1}) = N(x_t; √(1-β_t) x_{t-1}, β_t I)에서 β_t가 노이즈 스케줄입니다. 시간이 흐를수록 β_t가 커지면서 노이즈가 점점 강해지죠.

이제 **역방향 과정**, Reverse Process가 진짜입니다.

순수 노이즈 x_T에서 시작합니다. 모델에게 묻습니다. "이 노이즈가 살짝 덜 노이지한 상태, 즉 x_{T-1}은 어떻게 생겼니?" 모델이 답합니다. 한 단계 노이즈를 제거하죠. 또 묻습니다. "이건 어떻게 생겼니?" 또 답합니다. 또 한 단계. 이걸 T번 반복하면, 순수 노이즈에서 시작했음에도 불구하고 **완전히 새로운 이미지** x₀이 나옵니다.

여기서 직관이 좀 헷갈릴 수 있는데요, 핵심 통찰은 이거예요. **"순방향에는 학습이 없지만, 역방향에는 학습이 있다."** 모델은 학습 단계에서 수만 장의 이미지에 노이즈를 추가한 다음, "이 노이즈에서 원본은 어떤 방향이었는가"를 끊임없이 맞춰 보면서 노이즈 제거의 패턴을 익힙니다. 한 번도 본 적 없는 이미지도 만들 수 있는 이유는, **모델이 학습한 게 특정 이미지가 아니라 "노이즈에서 이미지로 가는 방향" 자체**이기 때문입니다.

*Hands-On Generative AI* 4장에서는 DDPM 논문 기반 코드를 1,000 스텝으로 돌립니다. GAN과 비교했을 때 Diffusion 모델의 가장 큰 단점이 바로 이 점이라고 짚어요. 추론에 1,000번의 forward pass가 필요하니까 GAN보다 훨씬 느립니다. 이걸 50~100스텝으로 줄이는 DDIM 같은 기법이 이후에 나온 이유죠.

기억할 두 단어로 정리하면 이렇습니다. **Forward는 노이즈 추가, Reverse는 노이즈 제거.** 그리고 모델이 학습하는 건 Reverse 방향의 한 스텝씩 노이즈 예측입니다.

### STEP 02 — Fine-tuning 전략: 적은 데이터로 특정 스타일 학습 (5분)

자, Diffusion이 어떻게 동작하는지는 알았습니다. 그런데 Stable Diffusion 같은 거대 모델을 처음부터 학습하는 건 우리 같은 일반 사용자에게는 불가능에 가깝습니다. 학습 데이터 수십억 장, GPU 수백 대, 학습 시간 수 주가 필요하거든요.

대안이 **Fine-tuning**입니다. 이미 학습된 모델을 우리 결함 이미지에 맞게 미세 조정하는 거죠. *Hands-On Generative AI* 7장에서 Fine-tuning 전략 세 가지를 정리합니다.

첫 번째가 **DreamBooth**입니다. 2022년 구글 리서치에서 나온 기법입니다. 단 **3~20장**의 특정 객체 이미지로 전체 모델을 업데이트하는 방식이에요. 한 인물의 얼굴, 한 제품의 모양 같은 걸 학습시키는 데 효과적이죠. 하지만 전체 모델을 업데이트하니까 시간이 수십 분 걸리고, GPU 메모리가 24GB 이상 필요합니다.

두 번째가 **LoRA**, Low-Rank Adaptation입니다. 오늘 실습에서 쓸 방법이에요. 원본 가중치 W는 완전히 고정해 두고, 그 옆에 **두 개의 작은 행렬 A와 B**를 추가합니다. 학습할 때는 ΔW = A × B만 업데이트하죠. 학습 끝나면 W + ΔW가 새 모델이 됩니다. 학습 파라미터가 전체의 0.1~1% 수준이라 **GPU 메모리도 8GB면 되고, 학습 시간도 수 분에서 30분 정도**로 끝납니다. 10~100장 정도면 특정 결함 패턴을 충분히 학습할 수 있어요.

세 번째가 **Textual Inversion**입니다. 모델 자체는 전혀 손대지 않고, 새로운 **텍스트 임베딩 토큰** 하나를 학습합니다. "이 토큰이 이런 스타일을 의미한다"고 모델에게 알려 주는 방식이죠. 가장 가볍지만 제어력이 제한적입니다.

세 가지를 한 표로 정리하면 이렇습니다.

| 방법 | 필요 이미지 | 학습 시간 | 파라미터 변경 |
|------|-----------|----------|------------|
| DreamBooth | 3~20장 | 수십 분 | 전체 모델 |
| LoRA | 10~100장 | 수 분 | 0.1~1% 추가 |
| Textual Inversion | 5~10장 | 수 분~수 시간 | 임베딩만 |

오늘 실습에서는 **사전에 fine-tuning이 완료된 LoRA 가중치**를 그냥 불러와서 씁니다. 학습 자체는 시간이 너무 오래 걸리니까, 결과만 가지고 생성 품질을 실험하는 방식이죠. 코드로 보면 이런 식입니다.

```python
from diffusers import StableDiffusionPipeline
import torch

pipeline = StableDiffusionPipeline.from_pretrained(
    "runwayml/stable-diffusion-v1-5",
    torch_dtype=torch.float16  # 반정밀도로 메모리 절약
)
pipeline.load_lora_weights("./lora_defect_weights")
pipeline = pipeline.to("cuda")
```

세 줄밖에 안 되죠. Hugging Face 허브에서 사전 학습 모델을 불러오고, 우리가 학습해 둔 LoRA 가중치를 얹고, GPU로 옮기는 게 전부입니다. 이 패턴이 표준이에요.

### STEP 03 — 생성 품질 평가: 무엇이 좋은 이미지인가 (3분)

마지막으로 한 가지. **생성된 이미지가 좋은지는 어떻게 판단하는가?**

가장 기본은 **육안 검사**입니다. 사람이 보고 "실제 결함처럼 보이는가"를 판단하는 거죠. 가장 직관적이지만 주관적이고, 대량 평가가 안 됩니다.

객관적 지표가 두 가지 있어요.

첫 번째가 **SSIM**, Structural Similarity Index Measure입니다. 두 이미지의 **구조적 유사도**를 0과 1 사이로 측정합니다. 1이면 완전히 동일, 0이면 완전히 다름. 단순히 픽셀 차이가 아니라 휘도, 대비, 구조 세 가지를 종합해서 판단합니다. **실무 기준으로 0.7 이상이면 유사한 이미지로 봅니다.** 결함 한 장 vs 결함 한 장 비교에 유용하죠.

두 번째가 **FID**, Fréchet Inception Distance입니다. SSIM이 한 장 대 한 장 비교라면, FID는 **이미지 집합 대 집합** 비교입니다. 생성한 이미지 200장의 분포와 실제 이미지 200장의 분포 사이의 거리를 측정해요. **낮을수록 좋고**, 10 미만이면 우수, 50 미만이면 양호로 봅니다. 단점은 신뢰할 수 있는 FID를 계산하려면 **최소 수백 장**이 있어야 한다는 거예요.

코드는 간단합니다. SSIM은 scikit-image에 들어 있어요.

```python
from skimage.metrics import structural_similarity as ssim
import numpy as np

def calculate_ssim(real_img, generated_img):
    real_gray = np.mean(real_img, axis=2)
    gen_gray = np.mean(generated_img, axis=2)
    return ssim(real_gray, gen_gray, data_range=1.0)
```

오늘 실습에서는 주로 SSIM을 씁니다. FID는 수백 장이 필요하니까 강의 시간 안에는 안 맞고, 심화 자료로 코드만 첨부했습니다.

세 가지를 같이 쓰는 게 정석입니다. **육안으로 거르고, SSIM으로 한 장씩 점검하고, FID로 전체 분포를 검증하는** 순서로요.

## 1-3. Claude Code 시연 (10분)

저는 이런 프롬프트로 시작하겠습니다.

```
Stable Diffusion으로 제조 결함 이미지를 생성하고 품질을 평가해줘.
- diffusers 라이브러리로 파이프라인 구성
- 결함 유형 3가지 프롬프트:
  "scratch defect on metal surface, industrial quality inspection"
  "bubble defect on painted surface, factory inspection photo"
  "crack defect on ceramic component, quality control"
- num_inference_steps=50, 유형당 3장
- 결과: 3x3 그리드 시각화
- 생성 이미지와 합성 실제 결함의 SSIM 계산
- GPU 없으면 CPU fallback
```

*(시연 진행)*

Claude Code가 코드를 만드는 동안 한 줄씩 같이 따라가 보세요.

먼저 **파이프라인 로드**입니다. `StableDiffusionPipeline.from_pretrained`로 모델을 가져옵니다. 처음 실행 시 수 GB의 가중치를 받느라 시간이 좀 걸리는데, 캐시에 저장되니까 두 번째부터는 즉시 로드되죠. `torch_dtype=torch.float16`은 메모리 절약을 위한 반정밀도 옵션이에요. GPU에서만 의미가 있고, CPU에서는 float32로 떨어뜨립니다.

다음이 **결함 유형 3가지 생성**입니다. 세 가지 프롬프트를 차례로 돌리면서 각각 3장씩, 총 9장의 이미지를 생성합니다. `num_inference_steps=50`은 우리가 이론에서 봤던 "역방향 노이즈 제거"를 50번 반복한다는 뜻이죠. 1000번이 정석이지만 DDIM 같은 빠른 샘플러를 쓰면 50번으로도 충분히 좋은 품질이 나옵니다.

세 번째가 **3x3 그리드 시각화**. `matplotlib`의 subplot으로 묶으면 한 화면에서 비교할 수 있어요. 같은 행은 같은 결함 유형, 같은 열은 같은 생성 인덱스. 같은 프롬프트에서도 시드가 다르면 결과가 미묘하게 다르다는 걸 시각적으로 보여주는 게 포인트입니다.

마지막으로 **SSIM 평가**. 시연에서는 시간 관계상 합성 노이즈 이미지를 "실제 결함"으로 가정하고 비교합니다. 진짜 실험에서는 라벨링된 실제 결함 한 장과 생성 결함 한 장을 비교하는 거죠.

자, 결과가 나왔습니다.

세 가지 결함 유형이 시각적으로 구분 가능한 패턴으로 나옵니다. 스크래치는 직선형, 버블은 원형, 크랙은 가지형으로요. 같은 프롬프트의 3장도 위치와 형태가 조금씩 다릅니다. **이게 Diffusion의 무작위성**입니다. 시드를 고정하지 않으면 매번 다른 결과가 나옵니다.

여기서 한 가지 흔한 문제를 짚을게요. **Mode Collapse**라고 들어 보셨나요?

시연 후 질문을 던지겠습니다. **"생성 이미지가 너무 비슷하게만 나오는 mode collapse는 왜 발생하고, 어떻게 해결하는가?"**

Mode collapse는 모델이 학습 데이터의 일부 패턴에만 빠져서 다양성을 잃어버리는 현상입니다. *Generative Deep Learning* 4장에서 GAN의 가장 유명한 실패 모드로 등장합니다. Diffusion에서도 fine-tuning 단계에서 학습 데이터가 너무 좁으면 비슷한 문제가 발생할 수 있어요.

원인은 보통 세 가지. **학습 데이터 다양성 부족**, **학습률 설정 실수**, **모델 용량 부족**입니다. 해결책은 **데이터 증강을 충분히 적용**, **프롬프트 다양화로 무작위성 주입**, **LoRA 가중치 조정** 같은 것들입니다. Diffusion의 경우 GAN보다는 mode collapse가 덜 일어나는 편이지만, fine-tuning 데이터가 10장 같은 극소량일 때는 여전히 주의해야 합니다.

다음으로 넘어가기 전 한 가지 더. 지금 보시는 생성 이미지가 100% 완벽하지는 않습니다. 텍스트가 흐릿하게 보이거나, 결함의 형태가 미묘하게 부자연스러울 수 있어요. 이런 한계는 섹션 2의 **프롬프트 제어**와 섹션 3의 **합성 데이터 검증**에서 보완해 나갑니다.

## 2-1. 문제 제기 (3분)

섹션 1의 프롬프트는 **"스크래치 결함 이미지"** 였습니다. 모델이 만들어 준 건 무작위 스크래치였죠. **어디에 났는지, 얼마나 깊은지, 어떤 방향인지** 우리가 제어하지 못합니다. 그냥 "스크래치가 들어 있는 이미지"가 나올 뿐이에요.

구체적인 예를 들어 볼게요. 스크래치 결함 중에서도 길이 1mm짜리, 5mm짜리, 10mm짜리가 골고루 필요합니다. 깊이도 표면 긁힘부터 깊은 홈까지 다양하게 있어야 하고, 방향도 수직, 수평, 대각선이 골고루 있어야 하죠. 그래야 학습한 모델이 어떤 종류의 스크래치든 잡아낼 수 있습니다.

실제 현장 데이터를 모으면 보통 **편향**이 있어요. 라인 특성상 수직 스크래치가 많이 나오고, 깊은 스크래치는 거의 없고, 1mm짜리 짧은 스크래치는 데이터에 안 잡힙니다. 모델이 그런 데이터로만 학습하면 본 적 없는 형태에는 약합니다.

"스테인리스 스틸 표면 왼쪽 가장자리에
 깊이 5mm의 수직 스크래치 결함,
 산업용 품질 검사 사진, 고해상도"

이런 식으로 텍스트 한 줄로 결함의 모든 속성을 지정하고, 그대로 이미지가 만들어진다면 데이터셋의 **빈 구멍을 정확히 채울 수 있죠**.

## 2-2. 이론 (13분)

### STEP 01 — 조건부 생성의 직관: CGAN에서 Diffusion으로 (4분)

조건부 생성, 즉 **특정 조건에 맞는 이미지를 만드는 기술**의 진화부터 짚고 가겠습니다. *Generative Deep Learning* 2판 4장과 13장에서 이 흐름을 정리합니다.

시작은 **CGAN**, Conditional GAN입니다. 2014년 Goodfellow 등이 GAN 원논문을 발표하고 같은 해에 등장한 변종이죠. CGAN은 일반 GAN에 **조건 c**를 추가합니다. Generator의 입력에 노이즈 z뿐만 아니라 클래스 레이블 c가 같이 들어가요. "스크래치"라는 레이블을 주면 스크래치 이미지를, "버블"이면 버블 이미지를 만들죠.

CGAN의 한계는 명확합니다. **조건이 클래스 레이블 수준**이라는 것. "스크래치"는 표현할 수 있어도 "왼쪽에 5mm 깊이의 수직 스크래치"는 표현이 안 됩니다. 클래스를 무한히 늘릴 수도 없고요.

2022년 이후 **Text-to-Image Diffusion**이 등장하면서 이 한계가 깨졌습니다. 조건이 클래스 레이블이 아니라 **자연어 문장 전체**로 확장된 거예요. Stable Diffusion, DALL·E 2, Imagen 같은 모델들이 이 흐름의 산물입니다.

핵심 아이디어는 **같습니다**. 조건부 생성이라는 발상은 변하지 않았어요. 다만 **조건을 표현하는 방식이 클래스에서 자연어로 진화**한 거죠. 자연어는 무한한 표현력을 가지고 있으니까, 결과적으로 무한한 제어가 가능해진 겁니다.

여기서 잠깐 GAN의 한계 하나를 더 짚을게요. *Generative Deep Learning* 4장에서 자세히 다루는 **Mode Collapse**입니다. Generator가 Discriminator를 속이기 가장 쉬운 한두 가지 패턴에만 빠져서 같은 이미지만 반복 생성하는 현상이죠. Diffusion은 학습 방식이 다르기 때문에 mode collapse가 훨씬 덜 일어납니다. 그래서 Diffusion이 GAN을 대체하는 흐름이 강해진 거예요.

우리가 오늘 쓸 도구는 Stable Diffusion입니다. 클래스 레이블이 아니라 **문장**으로 조건을 줄 수 있는 도구죠. 그 다리가 어떻게 만들어지는지, 다음 단계로 가 봅시다.

### STEP 02 — CLIP: 텍스트와 이미지를 연결하는 다리 (5분)

Stable Diffusion이 텍스트를 "이해"하는 비결은 **CLIP**이라는 모델입니다. *Generative Deep Learning* 13장에서 자세히 다루고, *Hands-On Generative AI* 2장에서도 핵심으로 소개됩니다.

CLIP, Contrastive Language-Image Pre-training. 2021년 OpenAI에서 공개한 모델이고, **인터넷에서 수억 개의 (이미지, 텍스트 캡션) 쌍**을 모아서 학습했습니다. 학습 방식이 흥미로워요. **대조 학습**, Contrastive Learning이라는 방식입니다.

학습할 때 한 배치에 N쌍의 (이미지, 텍스트)가 들어옵니다. 모델은 N개의 이미지와 N개의 텍스트를 각각 벡터로 변환하고, N×N개의 모든 쌍에 대해 유사도를 계산합니다. 그리고 **정답 쌍의 유사도는 높이고, 나머지는 낮추는 방향**으로 학습합니다.

결과적으로 CLIP은 **텍스트와 이미지를 같은 벡터 공간에 놓는 능력**을 얻게 됩니다. "빨간 자동차"라는 텍스트의 벡터와 실제 빨간 자동차 사진의 벡터가 가까이 위치하게 되는 거죠.

```
"a scratch on metal"  ↔  [스크래치 이미지의 벡터]
"a bubble defect"     ↔  [버블 이미지의 벡터]
"a crack on ceramic"  ↔  [크랙 이미지의 벡터]

→ 비슷한 의미 = 가까운 벡터
```

Stable Diffusion은 이 CLIP을 **텍스트 인코더**로 씁니다. 우리가 프롬프트를 입력하면 CLIP이 그걸 벡터로 바꾸고, 그 벡터가 Diffusion의 노이즈 제거 과정에 **조건으로 주입**됩니다. 노이즈를 제거할 때마다 "이 텍스트 벡터에 가까워지는 방향으로 가라"는 가이드가 들어가는 거죠.

코드로 보면 이렇습니다.

```python
from transformers import CLIPTextModel, CLIPTokenizer

tokenizer = CLIPTokenizer.from_pretrained("openai/clip-vit-base-patch32")
text_encoder = CLIPTextModel.from_pretrained("openai/clip-vit-base-patch32")

prompts = [
    "scratch defect on metal surface",
    "deep crack on ceramic material",
    "bubble defect on painted surface"
]

for prompt in prompts:
    inputs = tokenizer(prompt, return_tensors="pt")
    embeddings = text_encoder(**inputs).last_hidden_state
    print(f"shape: {embeddings.shape}")
    # 결과: [1, 토큰 수, 512]
```

embedding의 shape이 `[1, 토큰 수, 512]`라는 게 중요합니다. **각 토큰별로 512차원의 벡터**가 만들어진다는 뜻이죠. "scratch defect on metal surface"가 7~8개의 토큰으로 쪼개지면, 각 토큰별로 의미 벡터가 만들어지고, 이게 Diffusion의 cross-attention 메커니즘으로 이미지 생성에 영향을 줍니다.

CLIP의 또 한 가지 강점은 **zero-shot 일반화**입니다. 학습 데이터에 없는 단어 조합도 잘 처리합니다. "스테인리스 스틸 표면의 깊은 수직 스크래치"라는 조합이 학습 데이터에 정확히 있을 리는 없지만, CLIP은 각 단어의 의미를 조합해서 합리적인 벡터를 만들어내요. 우리가 자유로운 프롬프트로 결함을 설계할 수 있는 근거가 바로 여기에 있습니다.

### STEP 03 — 제조 결함 프롬프트 설계 원칙 (4분)

좋아요. CLIP이 텍스트를 이미지 공간으로 옮겨 준다는 건 알았습니다. 그렇다면 **어떻게 써야 좋은 결과가 나오는지** 보겠습니다. 프롬프트 엔지니어링의 영역이죠.

제조 결함 생성을 위한 **프롬프트 설계 공식**을 하나 제안하겠습니다.

```
[결함 유형] + [위치] + [심각도] + [재질/배경] + [촬영 스타일]
```

다섯 요소가 모두 들어가야 풍부한 결과가 나옵니다. 예시를 보겠습니다.

```
"a deep vertical scratch defect on the left edge
 of a stainless steel surface,
 industrial quality inspection photo,
 high resolution, studio lighting"
```

분해해 볼게요. **결함 유형**은 "deep vertical scratch defect", 깊은 수직 스크래치. **위치**는 "on the left edge", 왼쪽 가장자리. **재질**은 "stainless steel surface", 스테인리스 스틸. **촬영 스타일**은 "industrial quality inspection photo, high resolution, studio lighting", 산업용 검사 사진, 고해상도, 스튜디오 조명. 다섯 요소 중 네 가지가 명시적으로 들어가 있죠.

좋은 프롬프트와 나쁜 프롬프트를 비교해 보겠습니다.

| 나쁜 프롬프트 | 좋은 프롬프트 |
|------------|-------------|
| `defect` | `hairline crack defect, 2cm length, diagonal orientation, on white ceramic surface, macro photography` |
| `very very bad defect extremely damaged` | `severe deep crack defect, multiple branching lines` |

첫 번째 비교: "defect" 한 단어는 모델 입장에서 너무 모호합니다. CLIP이 무수히 많은 가능성 중에 무작위로 하나를 골라야 하니까 결과가 들쭉날쭉합니다. 반면 두 번째 쪽은 길이, 방향, 재질, 촬영 스타일까지 명시되어 있어서 결과가 일관성 있게 나옵니다.

두 번째 비교는 흔한 함정이에요. **강조를 반복한다고 효과가 커지지 않습니다.** "very very bad extremely damaged"는 CLIP이 보기엔 그냥 비슷한 의미가 중복된 노이즈입니다. 차라리 구체적인 단어, "severe deep crack with multiple branching lines"가 훨씬 효과적이죠.

그리고 한 가지 더 강력한 기법, **네거티브 프롬프트**입니다. 원하지 않는 요소를 명시하는 거예요.

```python
positive_prompt = "scratch defect on metal surface, inspection photo"
negative_prompt = "blurry, low quality, cartoon, illustration, text, watermark"

image = pipeline(
    prompt=positive_prompt,
    negative_prompt=negative_prompt,
    num_inference_steps=50,
    guidance_scale=7.5
).images[0]
```

네거티브 프롬프트는 "이런 요소는 빼 줘"라고 하는 거예요. 흐릿한 이미지, 만화풍, 일러스트, 텍스트, 워터마크 같은 게 들어가지 않게 하죠.

**주의할 점 하나**. 네거티브 프롬프트에 너무 많은 단어를 넣으면 오히려 품질이 떨어집니다. *Generative Deep Learning* 13장에서도 강조하는데, **5~7개 키워드가 적당**합니다. 그 이상은 모델에게 혼란만 주죠.

마지막으로 `guidance_scale=7.5`라는 매개변수. "프롬프트를 얼마나 강하게 반영할지"의 다이얼입니다. 낮으면 자유롭고 다양하지만 프롬프트와 다를 수 있고, 높으면 충실하지만 부자연스러울 수 있어요. 7.5가 기본 권장값이고, 실습에서 이걸 직접 바꿔 보면서 효과를 체감할 겁니다.

## 2-3. Claude Code 시연 (10분)

자, 이론을 코드로 옮길 시간입니다. 이번 시연의 포인트는 **프롬프트 설계가 생성 결과에 미치는 영향을 비교**하는 흐름입니다.

저는 이런 프롬프트로 시작하겠습니다.

```
제조 결함 유형별 프롬프트 템플릿을 설계하고 이미지를 생성해줘.

결함 유형별 프롬프트 (각각 3가지 강도):
1. 스크래치: 경미 / 중간 / 심각
2. 기포: 소형 / 중형 / 대형
3. 크랙: 표면 / 깊은 / 관통

- diffusers 파이프라인으로 각 프롬프트 생성
- 결과: 3x3 그리드 (행=결함유형, 열=심각도)
- guidance_scale=7.5 고정
- guidance_scale을 3 / 7.5 / 15로 바꾼 비교도 추가
```

*(시연 진행)*

먼저 **프롬프트 템플릿 설계**부터 봅니다. Claude가 다음과 같이 함수를 만들어 줄 거예요.

```python
def build_prompt(defect_type, severity, material="metal"):
    severity_map = {
        "light": "minor surface",
        "medium": "moderate",
        "heavy": "severe deep"
    }
    return (
        f"{severity_map[severity]} {defect_type} defect "
        f"on {material} surface, "
        f"industrial quality inspection photo, "
        f"high resolution, neutral background"
    )
```

이런 식으로 **결함 유형 × 심각도를 조합해서 자동으로 프롬프트를 만드는 함수**가 핵심이에요. 9가지 조합을 일일이 손으로 적는 대신 파라미터로 관리하는 거죠. 데이터셋을 체계적으로 만들 때 이런 함수가 표준입니다.

다음으로 **이미지 생성 루프**입니다. 9가지 조합 각각에 대해 한 장씩 생성합니다. 시드를 고정해서 각 조합의 차이를 명확히 비교할 수 있게 만들어요.

3x3 그리드로 시각화하면 한눈에 들어옵니다. 같은 행, 다른 열을 보면 같은 결함 유형이지만 심각도가 달라지는 게 보입니다. 다른 행을 보면 결함 유형 자체가 다르게 나오죠.

그리고 두 번째 실험. **guidance_scale 비교**입니다.

같은 프롬프트, 같은 시드, 다른 guidance_scale 세 가지. 3, 7.5, 15입니다.

3일 때는 결과가 가장 다양하지만 프롬프트와 다소 느슨하게 연결됩니다. 모델이 "프롬프트는 참고만 하고 자유롭게" 만드는 거죠. 7.5가 균형점이에요. 프롬프트도 잘 반영하고 자연스러운 이미지가 나옵니다. 15는 너무 강하게 반영해서 결과가 **과포화** 되거나 **아티팩트**가 생기기 시작합니다. 색이 너무 진하거나, 결함이 비현실적으로 강조되죠.

이 비교가 보여 주는 게 뭐냐면요. **guidance_scale은 너무 낮아도 안 되고 너무 높아도 안 됩니다.** 7.5 근처에 sweet spot이 있어요.

시연 후 질문을 던지겠습니다. **"동일한 프롬프트인데 결과가 매번 다른 이유는? 재현 가능한 결과를 얻으려면?"**

답은 단순합니다. Diffusion은 **시작 시점의 노이즈가 무작위**이기 때문에 같은 프롬프트라도 매번 다른 결과가 나옵니다. 시드를 고정하면 같은 노이즈에서 시작하니까 같은 결과가 나오죠.

```python
import torch
generator = torch.Generator(device="cuda").manual_seed(42)
image = pipe(prompt, generator=generator).images[0]
```

실무에서는 **데이터셋 재현성을 위해 시드를 항상 기록**해 두는 게 좋습니다. 어떤 결함 이미지가 어떤 시드에서 나왔는지 메타데이터로 같이 저장해 두면, 나중에 같은 이미지를 다시 만들 수 있어요.

## 3-1. 핵심 개념 + 실험 설계 (10분)

자, 마지막 섹션입니다. 가장 중요하면서도 가장 어려운 질문을 다룹니다.

**"우리가 만든 합성 데이터가 실제로 모델 성능을 올려 주는가?"**

섹션 1과 2에서 우리는 결함 이미지를 자유롭게 생성할 수 있는 능력을 얻었습니다. 그런데 이건 **수단**입니다. 진짜 목적은 결국 **탐지 모델의 성능 향상**이죠. 합성 데이터가 그 목적에 봉사하지 못한다면, 아무리 좋은 이미지를 만들어도 의미가 없습니다.

### 합성 데이터의 기회와 위험

*Practical Synthetic Data Generation* 1장에서 합성 데이터의 본질을 한 줄로 정의합니다. **"실제 데이터에서 학습한 분포의 통계적 성질을 공유하면서, 실제 개인을 식별할 수 없는 데이터."** 잘 만들면 매우 유용한 도구지만, 잘못 만들면 독이 됩니다.

**합성 데이터가 효과적인 시나리오** 세 가지를 보겠습니다.

첫째, **소수 클래스 보강**입니다. 결함처럼 애초에 희귀한 데이터를 합성으로 채우면, 모델이 결함 패턴을 충분히 학습할 수 있어요. **Recall이 올라갑니다.**

둘째, **다양성 확보**입니다. 실제 데이터에 없는 결함 조합, 예를 들어 "오른쪽 모서리에 10mm짜리 깊은 스크래치"를 합성으로 만들어 두면, 모델이 처음 보는 형태에도 강해집니다. **일반화 성능**이 개선되죠.

셋째, **라벨링 비용 절감**입니다. 합성 데이터는 생성 단계에서 이미 라벨이 결정되어 있어요. "이건 스크래치 결함을 만들어 달라"고 했으니까 그대로 스크래치 라벨이 붙는 거죠. 사람이 일일이 라벨링할 필요가 없습니다.

**반면 합성 데이터가 해로운 시나리오**도 분명히 존재합니다.

첫째, **생성 품질이 낮을 때**. SSIM이 0.3 같은 낮은 품질의 이미지를 학습에 넣으면 그건 노이즈를 주입하는 셈입니다. 모델이 잘못된 패턴을 외우게 됩니다.

둘째, **합성 비율이 너무 높을 때**. 합성이 80%, 90%가 되면 모델이 학습하는 분포가 실제 데이터 분포에서 점점 멀어집니다. 테스트 시점에 실제 이미지를 만나면 일반화가 안 됩니다.

셋째, **도메인 불일치**. 합성 이미지가 색상, 텍스처, 조명 같은 미묘한 부분에서 실제와 다르면, 모델이 그 부수적인 특징을 결함의 신호로 잘못 학습할 수 있습니다.

가장 흔한 실패 시나리오를 들어 볼게요. **합성 스크래치는 완벽하게 직선이고 깊이가 균일합니다. 실제 스크래치는 불규칙한 방향과 깊이 변화가 있습니다.** 합성으로만 학습한 모델은 "완벽한 직선 스크래치"만 인식하게 되고, 현장의 진짜 스크래치는 놓칩니다.

결론은 명확합니다. **"얼마나 섞을 것인가"가 핵심 질문**입니다. 0%도 아니고 100%도 아니고, 그 사이 어딘가에 최적점이 있어요.

### 실험 설계: 무엇을 비교할 것인가

좋은 답을 얻으려면 좋은 실험 설계가 필요합니다. *Practical Synthetic Data Generation* 4장의 평가 방법론을 따라가 보겠습니다.

데이터셋 구성은 이렇습니다.

```
실제 결함 이미지:  50장 (현실적으로 모은 소량)
합성 결함 이미지: 200장 (섹션 1·2에서 생성)
테스트셋:        20장 (실제, 절대 학습에 사용 안 함)
```

**테스트셋은 절대 학습에 안 들어갑니다.** 제가 가장 강조하고 싶은 원칙이에요. 합성 데이터의 효과를 검증하려면 평가는 항상 **실제 데이터**로 해야 합니다. 합성 데이터로 평가하면 "합성으로 학습한 모델은 합성을 잘 맞춘다"는 동어반복밖에 안 나옵니다.

비교할 데이터 구성 네 가지입니다.

- **Baseline**: 실제 50장만. 기준선.
- **실험 A (1:1)**: 실제 50장 + 합성 50장.
- **실험 B (1:3)**: 실제 50장 + 합성 150장.
- **실험 C (합성만)**: 합성 200장만. 극단적 케이스.

평가 지표는 세 가지를 봅니다.

- **F1-Score**: Precision과 Recall의 조화 평균. 균형 잡힌 성능.
- **Recall**: 결함을 놓치지 않는 능력. **제조 안전에서는 이게 가장 중요합니다.**
- **Precision**: 오탐 비율. 너무 낮으면 알람 피로가 발생합니다.

Session 1에서 배운 불균형 데이터 평가 지표가 여기서 다시 쓰입니다. 같은 도구를 다른 문제에 적용하는 거죠.

이 실험에서 우리가 보고 싶은 건 이거예요. **"실제만"이 가장 보수적이고, "합성만"이 가장 위험하고, 1:1 또는 1:3 어딘가에 최적점이 있을 가능성이 높다."** 결과가 어떤지는 직접 돌려 봐야 합니다.

## 3-2. Claude Code 시연 (15분)

자, 실험 설계는 됐으니 코드로 옮깁시다. 이번 시연의 핵심은 **코드를 만드는 것 자체가 아닙니다.** 코드는 Claude가 만들어 줍니다. 진짜 중요한 건 **나온 결과를 보고 "왜 이런 결과가 나왔는지" Claude와 대화하는 흐름**입니다.

저는 이런 프롬프트로 시작하겠습니다.

```
합성 데이터 추가 효과를 비교하는 실험 코드를 만들어줘.

실험 설정:
- 실제 결함 이미지: 50장 (numpy로 합성)
- 합성 결함 이미지: 200장 (품질이 약간 낮은 버전으로 시뮬레이션)
- 분류 모델: ResNet18 fine-tuning
- 4가지 데이터 구성: 실제만 / 1:1 / 1:3 / 합성만

결과 시각화:
1. 4가지 모델의 Confusion Matrix (2x2 subplot)
2. F1 / Recall / Precision 비교 막대그래프
3. 합성 비율 vs 각 지표 꺾은선 그래프

GPU 없는 환경에서도 실행 가능하게 (간단한 CNN으로 대체)
```

*(시연 진행)*

코드가 만들어지는 동안 흐름을 짚을게요.

먼저 **데이터셋 시뮬레이션**. 실제 결함과 합성 결함을 numpy 배열로 만들어 둡니다. 실제는 일정한 패턴에 약간의 노이즈, 합성은 같은 패턴이지만 조금 더 단순하고 균일하게 만듭니다. 이 차이가 모델 학습에 어떻게 영향을 주는지가 우리가 보고 싶은 부분이죠.

다음으로 **모델 학습 함수**. ResNet18 또는 간단한 CNN을 fine-tuning합니다. 같은 학습 설정으로 4번 학습시키고, 각각의 결과를 저장합니다. 결정적 요인이 데이터 구성이라는 점을 명확히 하기 위해 다른 변수는 모두 고정합니다.

세 번째 **평가**. 각 모델을 동일한 실제 테스트셋 20장으로 평가합니다. confusion matrix를 만들고, F1, Recall, Precision을 계산합니다.

자, 결과가 나왔습니다. 한번 보세요.

| 구성 | F1 | Recall | Precision |
|------|-----|--------|-----------|
| 실제만 | 0.72 | 0.65 | 0.81 |
| 1:1 | 0.78 | 0.82 | 0.75 |
| 1:3 | 0.74 | 0.88 | 0.64 |
| 합성만 | 0.55 | 0.71 | 0.45 |

(가상의 결과 예시입니다. 실제 숫자는 머신에서 다르게 나옵니다.)

여기서 가장 주목할 패턴이 있습니다. **합성 데이터를 더할수록 Recall은 올라가는데 Precision은 내려갑니다.** 1:1, 1:3에서 그 경향이 명확합니다. 그리고 합성만 쓰면 둘 다 떨어집니다.

이게 무슨 의미일까요? **Claude에게 결과 해석을 요청합니다.**

```
합성 데이터를 추가했더니 Recall은 올라갔는데 Precision은 내려갔어.
왜 이런 일이 생기는지 설명해 주고,
이 상황에서 어떤 지표를 더 믿어야 하는지 알려줘.
```

답을 정리해 보면 이렇습니다.

**Recall이 올라간 이유**: 합성 데이터가 모델에게 더 많고 다양한 결함 패턴을 보여 줍니다. 결과적으로 모델이 결함을 잘 찾아내는 능력이 올라가요. "이런 형태도 결함이다"는 폭이 넓어진 거죠.

**Precision이 내려간 이유**: 합성 결함과 실제 결함 사이에 미묘한 차이가 있고, 모델이 그 차이까지 "결함의 신호"로 학습합니다. 결과적으로 정상 이미지에서 합성 결함 비슷한 부수 특징이 보이면 결함이라고 잘못 판정합니다. 오탐이 늘어요.

**어떤 지표를 믿을지**는 **응용 분야에 따라 다릅니다**. Session 2에서도 강조했지만, **제조업의 안전 검사에서는 Recall이 압도적으로 중요**합니다. 결함을 놓치면 불량품이 출하되거나 안전 사고로 이어집니다. 반면 오탐은 인력으로 재검사하면 됩니다. 비용은 들지만 회복은 가능하죠.

그래서 결론적으로 우리 시나리오에서는 **1:1 혼합이 최적에 가까워 보입니다.** F1과 Recall이 둘 다 가장 좋고, Precision도 너무 떨어지지 않았으니까요. 1:3은 Recall은 더 좋지만 Precision이 너무 낮아져서 알람 피로가 우려됩니다. 합성만은 명백히 안 됩니다.

이 결론은 시드와 데이터에 따라 다를 수 있으니, **본인이 직접 돌려서 비슷한 패턴이 보이는지 확인**하는 게 중요합니다.

## 3-3. 자유 실습 (25분)

자유 실습입니다. 세 가지 난이도로 준비했어요. 본인 수준에 맞춰 골라서 진행하시고, 모두 다 하려고 하지 말고 **하나를 깊게** 파는 걸 추천합니다.

### 기본 — 합성 데이터 비율 최적점 찾기

가장 자연스러운 다음 질문은 "그러면 최적 비율은 정확히 얼마인가"입니다. 시연에서는 1:0, 1:1, 1:3, 합성만 네 가지였지만, 더 촘촘히 봐야 정확한 sweet spot을 찾을 수 있죠.

1. 합성 데이터 비율을 **0%, 25%, 50%, 75%, 100%** 다섯 가지로 변화시키며 모델을 학습합니다.
2. 각 비율에서 F1-Score, Recall, Precision을 기록합니다.
3. 비율 vs 각 지표의 꺾은선 그래프를 그립니다.

제출: **비율별 성능 표 + 최적 비율과 그 이유를 한 문단으로**.

여기서 한 가지 힌트. 비율이 올라갈수록 Recall은 단조 증가하다가 어느 시점부터 꺾이고, Precision은 단조 감소합니다. F1은 그 둘의 균형이라 종 모양 곡선이 나올 가능성이 큽니다. 최적점은 보통 25~50% 사이에 있어요.

### 중급 — 합성 이미지 품질 필터링

기본을 끝낸 분은 한 단계 더 가 봅시다. **모든 합성 이미지가 동일하게 가치 있지는 않습니다.** 품질이 낮은 합성을 걸러내면 같은 양으로 더 좋은 성능을 얻을 수 있을까?

1. 합성 이미지 200장 전체에 대해 SSIM을 계산합니다 (실제 결함 한 장과의 유사도).
2. 세 가지 필터링을 비교합니다.
   - 전체 사용
   - SSIM > 0.5만 사용
   - SSIM > 0.7만 사용
3. 각각의 데이터로 학습해서 성능을 비교합니다.

제출: **SSIM 임계값별 사용 이미지 수 + 성능 표 + 한 문단의 결론**. 그리고 Claude에게 한 가지 질문을 던져 보세요: **"FID와 SSIM 중 어떤 지표가 학습 성능을 더 잘 예측하는가?"**

이 질문에는 정답이 없습니다. FID는 분포 수준 평가라 학습 일반화와 더 잘 연결될 수 있고, SSIM은 개별 이미지 평가라 노이즈 필터링에 더 적합할 수 있어요. Claude의 답을 본인의 실험 결과와 비교해 보는 게 핵심입니다.

### 심화 — 결함 유형별 합성 효과 차이

가장 흥미로운 질문일 수 있어요. **합성이 모든 결함 유형에 동등하게 효과적인가?**

직관적으로는 그렇지 않을 가능성이 큽니다. 스크래치는 단순한 직선 패턴이라 합성이 쉽고, 크랙은 가지 모양이라 합성이 더 어렵습니다. **합성 품질이 높은 유형에서 학습 효과도 더 좋다**는 가설이 자연스럽죠.

1. 스크래치, 버블, 크랙 세 가지 결함 유형 각각에 대해 **실제만 학습 vs 실제+합성 학습**의 Recall 개선폭을 측정합니다.
2. 각 유형의 평균 SSIM(생성 품질)도 측정합니다.
3. **SSIM(x축) vs Recall 개선폭(y축) 산점도**를 그립니다.

제출: **산점도 + 상관관계 해석 한 문단**.

가설이 맞다면 양의 상관관계가 나올 겁니다. 안 맞다면 왜 그런지 추가로 분석해 보세요. 결함 유형의 본질적 복잡도? 데이터 양? 다른 요인?

## 클로징 — 전체 시리즈 마무리 (5분)

자, 오늘 Session 4에서 우리가 한 일을 먼저 정리하고, 그다음 네 세션 전체를 같이 돌아보겠습니다.

### Session 4 정리

**섹션 1**에서는 **Diffusion Model의 원리**를 직관으로 잡고 Stable Diffusion으로 결함 이미지를 생성했습니다. Forward는 노이즈 추가, Reverse는 노이즈 제거. 모델이 학습하는 건 노이즈 제거의 방향성이고, 그 능력으로 한 번도 본 적 없는 새 이미지를 만들 수 있다는 게 핵심이었죠. DreamBooth, LoRA, Textual Inversion 세 가지 Fine-tuning 전략을 비교하고, LoRA가 우리 같은 일반 사용자에게 현실적인 선택이라는 결론에 도달했습니다.

**섹션 2**에서는 **프롬프트로 결함을 정밀하게 제어**하는 방법을 배웠습니다. CLIP이 텍스트와 이미지를 같은 벡터 공간에 놓는 능력이 모든 Text-to-Image 생성의 기반이라는 것, 프롬프트 설계 공식 ([결함 유형] + [위치] + [심각도] + [재질] + [촬영 스타일])을 따라가면 일관성 있는 결과가 나온다는 것, 네거티브 프롬프트와 guidance_scale의 적절한 활용까지 다뤴습니다. 시드를 고정하면 재현 가능한 결과를 얻을 수 있다는 실무 팁도 같이 봤죠.

**섹션 3**에서는 가장 중요한 검증을 했습니다. **만든 데이터가 실제로 도움이 되는가.** 합성 비율 1:1, 1:3, 합성만의 네 가지 구성을 비교하면서, Recall과 Precision이 어떻게 움직이는지 데이터로 확인했죠. 결론은 단순합니다. **합성은 적당히 섞을 때 가장 효과적이고, Recall 중심 평가가 제조업에는 적합**하다는 것.

### 네 세션 전체를 한 흐름으로

여기서 한 단계 더 멀리 가 봅시다. 네 세션이 따로따로의 주제가 아니라 **하나의 흐름**으로 연결됩니다.

**Session 1: 센서 데이터 전처리.** 가장 기본기죠. Wavelet 노이즈 제거, 결측치 처리, 데이터 불균형, CNC 이상 탐지. 모든 ML 프로젝트가 여기서 출발합니다.

**Session 2: 딥러닝 이상 탐지와 예측.** 전처리된 데이터를 모델로 가져갑니다. 오토인코더, LSTM, NASA Turbofan 데이터로 RUL 예측. 시계열 데이터의 패턴을 학습하는 방법을 익혔죠.

**Session 3: Vision AI와 안전 모니터링.** 데이터 모달리티를 영상으로 확장했습니다. YOLO 객체 탐지, MediaPipe와 LSTM의 행동 인식, CCTV 통합 파이프라인. 시계열 모델링의 도구가 영상 도메인에서도 그대로 재활용된다는 걸 봤습니다.

**Session 4: 생성 AI와 합성 데이터.** 데이터 자체를 만드는 능력을 얻었습니다. Diffusion 원리, 프롬프트 제어, 합성 데이터 효과 검증.

이 네 가지를 **하나의 파이프라인**으로 연결하면 이렇게 됩니다.

```
부족한 결함 데이터
     ↓ Session 4: 생성 AI로 보강
충분한 학습 데이터
     ↓ Session 1: 전처리·라벨링·증강
정제된 데이터셋
     ↓ Session 2 & 3: 모델 학습
이상 탐지·RUL 예측·객체 탐지 모델
     ↓ Session 3: 실시간 파이프라인 구축
공장에 배포된 모니터링 시스템
```

이게 **제조 데이터 AI 실무의 전체 흐름**입니다. 데이터 확보부터 배포까지요. 각 세션이 이 흐름의 한 단계를 담당하는 거죠.
