### 3-A. `sec_appendix_classical_cv_background.md` — YOLO 이전의 세계

- **정체성**: YOLO 등장 전, 컴퓨터 비전이 물체를 어떻게 찾았는지(고전 CV ~ 2단계 검출기) 다루는 **배경 부록**. 본 강의 전 자율 읽기에 적합.
- **전달 흐름**: 용어 정리 → 연표 → 고전 CV(Sliding Window·Viola–Jones·HOG+SVM·DPM·특징점) → 딥러닝 2단계(R-CNN→Fast→Faster) → 1단계로 가는 다리(OverFeat·SSD·YOLO) → "고전 CV는 죽지 않았다"(최신 AI와 결합) → 요약.
- **핵심 메시지**: 딥러닝 혁명의 본질은 **"사람이 특징을 설계 → 기계가 특징을 학습"**으로의 대전환(2012년 전후). 그리고 고전 CV는 옛 기술이 아니라 **전처리·추적·SLAM·기하 계산에서 지금도 현역**.
- **강조 포인트**: DPM을 만든 Ross Girshick이 곧장 R-CNN으로 딥러닝 시대를 연 일화(시대의 다리). anchor·NMS·후보영역 같은 용어가 **여기서 태어나 YOLO에서 회수**된다는 점.
- **연결·복선**: 마지막 "한 번에 보자"가 YOLO 변천사의 첫 문장으로 그대로 이어집니다. ByteTrack(칼만 필터)은 AI City 실습에서 다시 등장.
- **권장 시간**: 사전 읽기 + 강의 중 10분 빠른 훑기.
- **예상 질문**: *"요즘 다 딥러닝인데 고전 CV를 왜 배우나요?"* → 엣지에서 가볍고, 해석 가능하고, 데이터 없이 동작하며, 기하 계산이 정확해 **전처리·후처리·추적에 지금도 박혀 있다**고 답해 주세요(ByteTrack·RANSAC·SLAM 예시).

### 3-B. `sec_appendix_yolo_evolution.md` — YOLO v1부터 YOLO26까지

- **정체성**: YOLO 세대별 변천사 + 심화(NMS·경량화) + 실무(라이선스·버전 선택)까지 묶은 핵심 부록.
- **전달 흐름**: 용어 → 한눈에 보는 표/타임라인 → 1~6세대 서사 → 심화(YOLOv8 vs v10, NMS가 실무에서 골칫거리인 이유, 경량화 4종) → 실무 후기(AGPL 라이선스, 최신≠최선) → 요약.
- **핵심 메시지**: **v8 → v10 → YOLO26의 흐름 = "정확도 경쟁"에서 "배포 친화성(후처리·속도) 경쟁"으로의 패러다임 이동.** anchor→anchor-free, NMS→NMS-free가 그 축.
- **강조 포인트**: ① NMS가 왜 실무에서 비싼가(순차 연산·export·재튜닝), ② 경량화는 **구조적 pruning만 실속**·"논문 sparsity ≠ 실제 latency", ③ **AGPL 라이선스 함정**(파인튜닝 가중치까지 적용), ④ **최신이 곧 최선은 아니다**(Ultralytics조차 프로덕션엔 YOLO11/26 권장).
- **연결·복선**: 고전 CV에서 심은 anchor·NMS가 여기서 회수. sec1 본문(YOLOv8 3등분 구조)과 직결. "배포 친화성" 명제는 AI City 실습에서 실증.
- **권장 시간**: 세대 흐름 15분 + 심화·라이선스 10분.
- **예상 질문**: *"실무에선 어떤 버전을 쓰나요?"* → 학습·시제품 YOLOv8/11, 엣지·저전력 YOLO26, 상업 제품은 **라이선스 먼저**(부담되면 YOLOX 등). *"AGPL이 진짜 문제 되나요?"* → 가중치·웹서비스까지 공개 의무가 걸리므로 상용은 Enterprise License 또는 대안 검토.

### 3-C. `sec_appendix_action_recognition_research.md` (+ `s3_2_action_model_landscape.svg`)

- **정체성**: sec2 실습(MediaPipe + LSTM)의 **배경 연구 동향**. 골격 기반 모델 계보(ST-GCN), 데이터 현실, 모델 지형도, 최근 챌린지 흐름.
- **전달 흐름**: ST-GCN 계보 → 대표 연구 사례 → "데이터 부족"이라는 현실 → 모델 지형도(SVG)와 운영 요건별 선택 → 한계·대응 → 공개 데이터셋 → 챌린지·낙상 검출 → 요약.
- **핵심 메시지**: 행동 인식엔 **YOLO 같은 단일 표준이 없다 → "운영 요건"으로 고른다.** 현장 도입은 거의 항상 **"사전학습 백본 → 자체 소량 데이터 fine-tune → 시계열 후처리"** 3박자로 수렴.
- **강조 포인트**: ① **도메인 지식 주입이 모델 키우기를 이긴다**(CoG-STGCN: 무게중심 특징). ② 골격 좌표는 **개인정보가 없어** 사내 데이터 활용 부담이 적다. ③ SVG 지형도로 "골격↔RGB × 엣지↔클라우드" 두 축을 한 장에 보여주기.
- **연결·복선**: sec2의 LSTM에서 한 단계 위(그래프를 직접 모델링하는 ST-GCN). "속도 = 안전"이라는 세션 3 주제와 직결.
- **권장 시간**: 10~15분(SVG 한 장으로 지형도 설명 후 선택 기준 강조).
- **예상 질문**: *"RGB와 Skeleton 중 뭘 쓰나요?"* → 지연·자원·개인정보·동작 입도로 결정, 안전 모니터링은 골격이 주류·필요 시 하이브리드. *"공개 데이터가 없으면요?"* → pose 추출 + 소량 라벨로 자체 데이터셋이 현실적 첫 단계.

### 3-D. `sec_practice_aicity_fisheye.md` + `.ipynb` — AI City Track 4(피시아이)

- **정체성**: 2025 AI City Challenge **Track 4(피시아이 도로 객체 탐지)** 보충 실습. md는 배경, ipynb는 우승팀 **SmartVision** 추론을 Colab에서 재현 + CCTV 영상 분석(추적·ROI 침입·시간 안정화)까지 확장.
- **전달 흐름**: (md) 대회·4개 트랙 → Track 4가 CCTV와 닮은 이유 → FishEye8K → 피시아이 전처리 선택 → 우승 전략. (ipynb) 모델 로드·크기 확인 → 샘플 추론 → 속도(FPS) 측정 → (TensorRT는 설명만) → 영상 파이프라인 → ByteTrack → ROI 침입 → 시간 안정화.
- **핵심 메시지**: 엣지 안전 탐지의 승부처는 **F1과 FPS를 동시에 만족하는 실용 파이프라인.** 우승 비결 = **(일반 사전학습) + (강한 교사 증류·의사라벨) + (타깃 도메인 합성 증강) + (런타임 최적화)**. 탐지 모델은 "위반"을 모른다 → **탐지 위에 추적+규칙+시간 조건**을 얹어 이벤트를 만든다.
- **강조 포인트**: dewarping을 **안 하고** 원본 왜곡에 학습하는 선택(시야·화질 보존), GFLOPs 지수 감점이 주는 "경량 강제", ByteTrack = **딥러닝 검출 + 고전 칼만**(세션 3 배경 회수).
- **연결·복선**: YOLO 변천사의 "배포 친화성" 명제를 실제 수치로 회수. 세션 4의 "합성 증강"과도 연결(우승팀이 VisDrone을 피시아이로 합성).
- **권장 시간**: 25~35분. **운영 메모**: Colab 런타임 GPU(T4) 필요, TensorRT export 셀은 Colab에서 실행하지 않고 설명만(버전 비호환).
- **예상 질문**: *"왜 왜곡을 안 펴나요?"* → 보정은 시야·화질 손실 trade-off, 상위권은 원본 학습 + 증강으로 왜곡 흡수. *"모델이 '위반'을 어떻게 아나요?"* → 모델은 객체만 탐지, 위반은 ROI·체류시간 **규칙 레이어**가 판정.

### 3-F. 현장 적용 시 모델·파이프라인 선택 기준 (세 자료 회수용)

세 부록·실습이 공통으로 던지는 실무 질문 — *"그래서 현장엔 뭘 올리나?"* — 을 한 곳에 모았습니다. 정확도(mAP)부터 보지 말고, 아래 축을 **순서대로 좁혀 가며** 후보를 거른 뒤 마지막에 정확도를 비교하도록 안내하면 좋습니다.

| 고려 축 | 핵심 질문 | 현장 가이드 | 근거 자료 |
|---|---|---|---|
| **배포 위치·HW** | 엣지인가 서버인가 | 엣지 → 경량(YOLOv8n·YOLO26·MobileNet) + 양자화 + TensorRT / 서버 → 정확도 우선(YOLOv8m+) | YOLO 변천사·AI City |
| **속도·지연** | 실시간인가(30FPS≈33ms), 카메라 몇 대를 한 GPU로? | mAP만 보지 말고 **F1·FPS·latency 동시에**. "속도 = 안전" | AI City(GFLOPs 감점)·행동 인식 |
| **사용 시나리오** | 객체 탐지냐 행동 인식이냐 / 실시간 경고·24시간 감시·사후 분석 중 무엇 | 행동 인식은 단일 표준이 없어 **운영 요건으로 고른다**(골격·RGB·하이브리드) | 행동 인식 동향 |
| **개인정보** | 픽셀을 밖으로 내보내도 되나 | 민감하면 **골격 기반**(좌표만, 개인정보 없음) + 엣지 처리 | 행동 인식 동향 |
| **데이터·라벨** | 맞는 공개셋이 있나 | 없으면 자체 수집 + 소량 라벨; **"사전학습 백본 → fine-tune → 시계열 후처리"** 3박자 | 행동 인식 동향 |
| **후처리·배포 난이도** | export·재튜닝 부담은 | **NMS-free(v10/26)** 가 유리, 경량화는 **구조적 pruning만 실속** | YOLO 변천사 |
| **라이선스** | 상용 제품/서비스에 넣나 | Ultralytics YOLO = **AGPL-3.0**(가중치·웹서비스까지 공개 의무) → Enterprise License 또는 **Apache 계열(YOLOX 등)** | YOLO 변천사 |
| **버전·성숙도** | 꼭 최신이어야 하나 | **"최신 ≠ 최선"** — 자료·생태계가 풍부하고 내 기기에서 안정적인 버전(YOLOv8/11, 엣지엔 YOLO26) | YOLO 변천사 |

> **한 줄로**: 현장 선택은 **하드웨어 → 속도 → 시나리오 → 개인정보 → 데이터 → 후처리 → 라이선스 → 성숙도** 순으로 후보를 좁히고, **정확도(mAP)는 이 필터를 통과한 모델끼리 마지막에 비교**합니다.

```
예상 질문: "정확도 제일 높은 모델 쓰면 되지 않나요?"
→ 엣지에서 안 돌거나(속도), 상용에 못 쓰거나(라이선스), 우리 데이터에 안 맞으면(시나리오·데이터) mAP 1등도 현장에선 0점. 그래서 정확도는 '마지막 비교 기준'이라고 짚어 주세요.
```

## 세션 4 — Generative AI 합성 데이터

**진행 순서 제안** (기존 본문 + 추가 자료 통합): `sec0_landscape`(추가·도입) → `sec1_diffusion_model`(기존 본문) → `sec2_defect_generation`(기존 본문) → `sec_appendix_diffusion_defect_deep_dive`(추가·심화 ①②) → `sec_practice_defect_generation`(추가·생성 실습 Part A/B) → `sec3_synthetic_data`(기존·합성 데이터 고도화) → `sec_practice_vision_track2_diag` + `h4_mvtec_anomaly_detection`(추가·기존·성능 검증 실습) → `sec_practice`(기존·실습 모음) → `sec4_advanced`(기존·심화 자료). 즉 **"왜 → 어떻게 만드나 → 정말 도움이 되나"** 순서.

### 4-A. `sec0_landscape.md` — 5년의 큰 그림

- **정체성**: 세션 4 **도입**. diffusion 기반 결함 생성의 5년 진화(Phase 0~5), 변화를 관통하는 5개 축, 흔한 오해, 생성 vs VLM 두 갈래.
- **전달 흐름**: 0-1 문제 제기(이미지엔 SMOTE가 안 통한다) → 0-2 이론(STEP01 6단계 진화 / STEP02 5개 축 / STEP03 "training-free ≠ 생성 불필요" 오해 / STEP04 생성 vs VLM) → 0-3 다음 섹션 안내.
- **핵심 메시지**: 합성은 *"더 그럴듯한 그림"*에서 *"원하는 결함을, 라벨까지 정렬된 채로, 학습 없이"*로 진화했다. **합성과 VLM은 경쟁이 아니라 보완**(픽셀 정밀·양산 고정 제품 ↔ 라벨 없는 신제품·논리적 이상).
- **강조 포인트**: STEP03의 오해 풀기가 이 섹션의 하이라이트 — "training-free"는 **생성기를 제품마다 재학습 안 해도 된다**는 뜻이지 합성을 안 한다는 뜻이 아니다.
- **연결·복선**: Session 1 SMOTE·세션 3 임계값 → 세션 4 생성으로 이어지는 흐름을 명시. 4-1~4-3 예고, "잘 만들면 도움/독" 균형은 4-3 실습에서 검증한다고 복선.
- **권장 시간**: 10~15분 도입.
- **예상 질문**: *"학습 없이 탐지(zero-shot)하면 생성은 필요 없지 않나요?"* → STEP03 그대로 답: 생성 쪽 training-free와 탐지 쪽 zero-shot은 별개이며, 픽셀 정밀·고신뢰 양산엔 여전히 합성이 유효.

### 4-B. `sec_appendix_diffusion_defect_deep_dive.md` — 심화 ①·②

- **정체성**: 결함을 "만들어내는" 두 방법의 이론 심화. **① 파인튜닝**(Textual Inversion·DreamBooth·LoRA)과 **② 텍스트 기반 생성**(CLIP, 텍스트+마스크). sec0과 실습 사이의 다리.
- **전달 흐름**: 왜 생성이 필요한가(1분 요약) → 합성이 특히 유효한 영역 → ① 파인튜닝(few-shot 함정·PEFT·이미지+마스크 쌍·"내 강아지와 뭐가 다른가") → ② 텍스트 생성(CLIP·텍스트는 '무엇'엔 강·'어디'엔 약·두 갈래·잘 생성하기) → 두 방법 비교.
- **핵심 메시지**: 가장 중요한 산출물은 **이미지 + 마스크(=라벨) 쌍.** 파인튜닝 기법은 "내 강아지 사진 길들이기"와 **도구는 같지만 목적·평가가 다르다**(탐지기 성능이 오르는가). 텍스트(무엇) + 마스크(어디)는 **함께 쓰는 두 레버**.
- **강조 포인트**: LoRA가 왜 실용적인가(본체 동결, 보조 모듈만), few-shot 과적합·망각, "**믿을 수 있게** 생성"이 진짜 과제(경계 자연스러움·텍스트 일치·마스크 정렬·필터링).
- **연결·복선**: **① = 실습 Part B(LoRA), ② = 실습 Part A(inpainting)**로 직결. sec0 Phase 3(텍스트+마스크)와 AnomalyDiffusion 회수.
- **권장 시간**: 15~20분.
- **예상 질문**: *"LoRA와 DreamBooth 차이는?"* → DreamBooth는 본체를 살짝 재학습(무겁고 충실), LoRA는 보조 모듈만(가볍고 실용). *"몇 장으로 되나요?"* → 5~20장 수준, 단 과적합·망각 주의.

### 4-C. `sec_practice_defect_generation.ipynb` — 합성 결함 생성 실습

- **정체성**: 심화 **①·②에 1:1 대응**하는 실습. **Part A(② 텍스트+마스크 inpainting, 무학습)** → **Part B(① LoRA 파인튜닝)**. Hugging Face `diffusers`, 무료 Colab T4.
- **전달 흐름**: (A) 정상 이미지 → 마스크(=라벨) 만들기 → 텍스트+마스크로 결함 생성 → 정상/마스크/결함 비교. (B) 결함 5~10장 준비(MVTec AD·VisA) → 공식 LoRA DreamBooth 스크립트 → 데모 학습 → 학습한 LoRA로 우리 결함 생성. 마무리에서 두 방법 매핑.
- **핵심 메시지**: 두 파트 모두 목적은 **이미지 + 마스크(라벨)** 생산. Part A는 가볍고 즉석, Part B는 무겁지만 우리 도메인에 충실. **실무는 둘을 결합**(파인튜닝한 모델에 텍스트+마스크로 지시).
- **강조 포인트**: **마스크가 곧 세그멘테이션 정답 라벨** → 비싼 수작업 라벨링 절감. Part B의 새 토큰(`sks`)으로 "우리 결함"을 이름표 붙여 가르치는 개념.
- **연결·복선**: 심화 ①②의 실전. 둘을 한 번에 하는 연구급 코드로 AnomalyDiffusion 안내. 다음 실습(DIAG)에서 "생성이 실제로 성능을 올리나"로 이어짐.
- **권장 시간**: Part A 15분 + Part B 25~35분. **운영 메모**: Colab GPU(T4) 필수, Part B는 데모 설정(400 steps)이라도 무거우니 탭 유지 안내.
- **예상 질문**: *"왜 마스크가 라벨인가요?"* → 결함 위치를 픽셀로 표시한 것이 곧 세그멘테이션 정답. *"Part A·B 중 뭐부터?"* → 쉬운 A(무학습)로 감 잡고 B(학습)로 심화.

### 4-D. `sec_practice_vision_track2_diag.md` + `.ipynb` — VISION Track 2 / DIAG (성능 검증)

- **정체성**: VISION Challenge **Track 2(Data Generation)** 과제를 틀로 삼아, 공개 코드 **DIAG**(training-free, SDXL inpainting)로 **"합성이 실제 AP를 올리는가"**를 실험하는 실습. md는 배경(KSDD2·VISION·상위 솔루션·DIAG), ipynb는 생성→학습→AP 비교.
- **전달 흐름**: (md) KSDD2 데이터 → VISION 대회·두 트랙 → 상위 솔루션 DCDGANc → DIAG(무학습·in-distribution)·DCDGANc와 비교. (ipynb) DIAG 코드 클론 → KSDD2 준비 → 합성 전략 이해 → SDXL 생성 → **Baseline vs DIAG 증강 AP 비교** → 정리.
- **핵심 메시지**: 채점 기준은 **"네가 만든 데이터가 실제 테스트셋 성능(AP)을 올리는가."** DIAG는 **in-distribution**(정상 위 지정 영역만 변경)이라 "현실에 없는 결함" 위험이 작다. **평가는 항상 실제 테스트셋으로**(합성으로 평가하면 동어반복).
- **강조 포인트**: 학습형 GAN(DCDGANc) vs 무학습 diffusion(DIAG)의 철학 대비. 실습은 **양성 샘플이 없는 시나리오**(논문 보고 +28%)를 보이며, 이것이 "합성이 데이터를 처음부터 만들어 준다"는 메시지의 실증.
- **연결·복선**: 심화 ②의 "텍스트 + 마스크"의 실전판. sec0 0-3이 예고한 **4-3(합성 데이터 고도화)** 실습. Session 1 불균형 회수.
- **권장 시간**: 30~40분. **운영 메모**: Colab GPU 필수, SDXL 약 7GB 다운로드·생성이 느리므로 데모 설정(`IMGS_PER_PROMPT` 작게) 유지. KSDD2 다운로드가 막히면 공식 페이지에서 받아 `/content/ksdd2`에 풀도록 안내.
- **예상 질문**: *"VISION Track 2를 그대로 재현하는 건가요?"* → 과제 **정의를 틀로 삼고** KSDD2+DIAG로 채운 것(정확한 동일 재현은 아님, 노트북에 명시). *"Baseline은 결함이 0인데 학습이 되나요?"* → 양성 샘플이 전혀 없을 때를 보이는 의도된 설계로, **합성이 데이터를 만들어 줘야 탐지가 가능**함을 드러내는 비교.

## 마무리 — 마지막에 회수할 한 문장

> **세션 3**: "현장 점수를 가르는 건 mAP가 아니라 **F1과 FPS를 동시에 만족하는 실용 파이프라인.**"
> **세션 4**: "데이터가 없으면 끝이 아니다 — **없으면 만든다.** 단, 예쁜 가짜가 아니라 **라벨(마스크)까지 맞은 합성 데이터로, 실제 테스트셋 성능이 오를 때만** 진짜다."

세션 4를 한 번 더 풀어 두면: 제조 결함은 너무 드물어 "데이터가 없어서 못 한다"가 현실입니다. 세션 4의 발상 전환은 **그 빈자리를 생성으로 채우는 것** — 하지만 합성물의 합격 기준은 "보기 좋음"이 아니라 단 둘입니다. ① **결함 위치 라벨(마스크)까지 정렬**돼 있는가, ② 그걸로 학습한 탐지기가 **실제 테스트셋에서 더 잘 잡는가**. DIAG 실습의 *Baseline vs 합성 증강 AP 비교*가 바로 이 심판대입니다.

두 문장 모두 **"논문에서 끝나는 모델 vs 현장에서 사는 모델"**이라는 같은 질문의 두 얼굴입니다 — 세션 3은 *모델을 현장 제약(속도)에 맞추고*, 세션 4는 *부족한 데이터를 만들어 현장 성능을 끌어올린다*는 점만 다릅니다. 마지막 슬라이드에서 이 한 줄로 네 세션을 묶어 주시면 좋겠습니다.
